# 実装カタログ

bq-first パイプラインの実装物カタログ (題材: 不動産ハイブリッド検索)。`docs/README.md` §1 の 03 仕様 (`ディレクトリ / ファイル(一言コメント付き) / DB テーブル・リレーション / API エンドポイント / UI ページ / GCP リソース / Terraform を逐一掲載`) に章を 1:1 対応させる。スコープ確定は `docs/02_移行ロードマップ.md`。

Phase 0 (tfstate バケット作成・API 有効化) は Terraform の外側で手動実施する前提。本書は **Phase 10d + scripts/tests 再編 + setting.yaml 集約 + deploy-all/destroy-all による PDCA 完走検証後** (2026-04-20 時点) の最終状態を記す。旧 California Housing コード / Dataform / 監視 SQL / Cloud Run Service `ml-api` は全て削除済 (`ml-api` は `search-api` に rename)。

現状の `make check` 合格数: **183 tests PASS** (新規 3: dataform parity)。

**実 GCP `mlops-dev-a` での end-to-end 検証状況** (2026-04-20 実測):

| 検証項目 | 結果 |
|---|---|
| `make deploy-all` (空状態 → フル provisioning) | ✓ 成功。8 step 完走 ~12-15 分。WIF pool/provider の soft-delete 回復ロジック含む |
| `make destroy-all` (フル破壊) | ✓ 成功。54 リソース destroy ~3-5 分、tfstate バケットは保持 |
| PDCA 2 周目 (destroy 後の再 deploy-all) | ✓ 成功。WIF auto-recovery が動作、0 手介入で完走 |
| `make seed-minimal` (5 物件 test seed) | ✓ `properties_cleaned` / `property_features_daily` / `property_embeddings` に各 5 行 insert |
| `make ops-livez` | ✓ `{"status":"ok"}` |
| `make ops-search` | ✓ 5 物件返却、`me5_score` 付き、`final_rank == lexical_rank` / `score=null` (Phase 4 rerank-free の期待挙動) |
| `make ops-ranking` | ✓ top 5 の lexical/final/score/me5_score 整形出力 |
| `make ops-feedback` | ✓ `{"accepted": true}` (/search → /feedback round-trip) |
| `make ops-label-seed` | ✓ `posted=3` (click / favorite / inquiry 各 1 件 publish) |

**検証で判明 + 恒久対処した 7 件のギャップ** (全てコードベースに fix 込):

1. `terraform destroy` の `deletion_protection=true` ブロック → `enable_deletion_protection` var + `scripts/setup/destroy_all.py` で `-target=<7 BQ tables>` 経由の state-flip apply → destroy の 2 段構成
2. 半 destroy 後の SA / Dataset IAM orphan → `-target` で apply 範囲を絞り recreate を回避
3. WIF pool/provider 30 日 soft-delete → `scripts/setup/deploy_all.py::_recover_wif_state` が undelete + `terraform import` を自動実行
4. Cloud Run `ModuleNotFoundError: sentence_transformers` → `common/pyproject.toml` に `sentence-transformers>=3.0` 追加 (両 workspace 共有)
5. `sa-api` が `feature_mart` 権限不足で /search が 403 → `infra/modules/data/main.tf` に `api_feature_viewer` (bigquery.dataViewer) 追加
6. `terraform apply` が Cloud Run Service の env を reconcile して `ENABLE_SEARCH=true` を消す → `infra/modules/runtime/main.tf` の `lifecycle.ignore_changes` に `template[0].containers[0].env` を追加 (Service / Job 両方)
7. Cloud Run `/healthz` が GFE に予約されて HTML 404 → app に `/livez` alias を register、ops script は `/livez` を叩く

## 4. API エンドポイント

| メソッド | パス | 役割 |
|---|---|---|
| GET | `/livez` | 常時 OK (liveness)。**Cloud Run の Knative frontend が literal `/healthz` を予約**しており HTML 404 を返すので、デプロイ後の疎通確認には `/livez` を使う。`/healthz` も alias として register されているので local `make api-dev` 時はどちらでも届く |
| GET | `/readyz` | `candidate_retriever` がロード済か (readiness、503 if null)。成功時 `{status, search_enabled, rerank_enabled, model_path}` を返す |
| POST | `/search` | `SearchRequest` → `SearchResponse`。booster=None (Phase 4 fallback) なら `final_rank=lexical_rank` / `score=None`。booster ロード済なら `_score_candidates` で rerank + `score` を付与 |
| POST | `/feedback` | `FeedbackRequest` → `FeedbackResponse`。Pub/Sub 経由で `mlops.feedback_events` へ |
| POST | `/jobs/check-retrain` | 再学習条件評価 → 条件成立時 `retrain-trigger` へ publish。ranker 指標 (NDCG / feedback rows) のみを見る |
| POST | `/events/retrain` | Eventarc 目的地。training-job を起動 |

Pydantic schema: `SearchFilters` / `SearchRequest` / `SearchResultItem` (`score` は rerank 時のみ non-null) / `SearchResponse` / `FeedbackRequest` / `FeedbackResponse`。
