# 実装カタログ

Vertex AI 移行版 (`study-hybrid-search-vertex`) の実装物カタログ (題材: 不動産ハイブリッド検索)。`docs/README.md` §1 の 03 仕様 (`ディレクトリ / ファイル(一言コメント付き) / DB テーブル・リレーション / API エンドポイント / UI ページ / GCP リソース / Terraform を逐一掲載`) に章を 1:1 対応させる。スコープ確定は `docs/02_移行ロードマップ.md`。

Phase 0 (tfstate バケット作成・API 有効化) は Terraform の外側で手動実施する前提。本書は **Phase 3 `study-hybrid-search-cloud` をベースに Vertex AI レイヤを後付けし、Cloud Run Jobs 削除 + Vertex Feature Group/Endpoint 宣言 + Phase 10 deploy workflows 整備まで完了した 2026-04-21 時点** の状態を記す。Phase 3 相当の California Housing コード / 旧 `ml-api` はすべて削除済。

現状の `pytest` 合格数: **224 tests PASS** (Phase 9/10 の再編で `/events/retrain` 系 + 旧 `CloudRunJobRunner` adapter の test を retire、Vertex Experiments dual-write の lockstep test を追加)。

**実 GCP `mlops-dev-a` での end-to-end 検証状況**:

| 検証項目 | 結果 |
|---|---|
| Phase 3 (`study-hybrid-search-cloud`) 時点の PDCA 疎通 | ✓ `make deploy-all` / `destroy-all` × 2 周、5 ops-* 全 PASS (移行元リポで完走済) |
| Phase 4 (`study-hybrid-search-vertex`) 時点 | ローカル `pytest` 224 passed / `scripts.checks.layers` 30 files clean / `terraform validate` Success まで。**実 GCP での Vertex Endpoint / Feature Group / Pipeline 投入は未実施** (後日まとめて検証) |

**Phase 3 時点で検証済 + 恒久対処した 7 件のギャップ** (Vertex 移行後も有効、全てコードベースに fix 込):

1. `terraform destroy` の `deletion_protection=true` ブロック → `enable_deletion_protection` var + `scripts/setup/destroy_all.py` で `-target=<BQ tables>` 経由の state-flip apply → destroy の 2 段構成
2. 半 destroy 後の SA / Dataset IAM orphan → `-target` で apply 範囲を絞り recreate を回避
3. WIF pool/provider 30 日 soft-delete → `scripts/setup/deploy_all.py::_recover_wif_state` が undelete + `terraform import` を自動実行
4. `sa-api` が `feature_mart` 権限不足で /search が 403 → `infra/modules/data/main.tf` に `api_feature_viewer` (bigquery.dataViewer) 追加
5. `terraform apply` が Cloud Run Service の env を reconcile して `ENABLE_SEARCH=true` を消す → `infra/modules/runtime/main.tf` の `lifecycle.ignore_changes` に `template[0].containers[0].env` を追加 (Vertex 移行後は `VERTEX_*_ENDPOINT_ID` も同様に守られる)
6. Cloud Run `/healthz` が GFE に予約されて HTML 404 → app に `/livez` alias を register、ops script は `/livez` を叩く
7. Cloud Run image の reconcile → `lifecycle.ignore_changes = [... image ...]` で `make deploy-*-local` / CI が push する real image が `terraform apply` に巻き戻されない

### 2.3 `jobs/src/` (Python パッケージ `training` + `meili_sync`)

## 4. API エンドポイント

| メソッド | パス | 役割 |
|---|---|---|
| GET | `/livez` | 常時 OK (liveness)。**Cloud Run の Knative frontend が literal `/healthz` を予約**しており HTML 404 を返すので、デプロイ後の疎通確認には `/livez` を使う。`/healthz` も alias として register されているので local `make api-dev` 時はどちらでも届く |
| GET | `/readyz` | `candidate_retriever` + `encoder_client` がロード済か (readiness、503 if null)。成功時 `{status, search_enabled, rerank_enabled, model_path}` を返す |
| POST | `/search` | `SearchRequest` → `SearchResponse`。`VertexEndpointEncoder.embed(query, "query")` で query vector 取得 → `BigQueryCandidateRetriever` で RRF 融合候補取得 → `reranker=None` なら `final_rank=lexical_rank` / `score=None`、`VertexEndpointReranker` 指定時は `_score_candidates` で rerank + `score` を付与 |
| POST | `/feedback` | `FeedbackRequest` → `FeedbackResponse`。Pub/Sub 経由で `mlops.feedback_events` へ |
| POST | `/jobs/check-retrain` | 再学習条件評価 → 条件成立時 `retrain-trigger` へ publish。ranker 指標 (NDCG / feedback rows) のみを見る |

`/events/retrain` は Phase 9 で削除済 (Eventarc subscriber は `functions/pipeline_trigger` Cloud Function に切替、search-api 側から training-job を起動する経路は無くなった)。

Pydantic schema: `SearchFilters` / `SearchRequest` / `SearchResultItem` (`score` は rerank 時のみ non-null) / `SearchResponse` / `FeedbackRequest` / `FeedbackResponse`。
