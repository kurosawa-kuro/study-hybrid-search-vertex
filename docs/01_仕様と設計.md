# 仕様と設計

## 共通前提

- **目的**: GCP 上での MLOps パイプライン構築の技術検証・ベストプラクティス確立
- **題材**: 不動産ハイブリッド検索 (クエリ文 + フィルタ → 物件ランキング上位 N 件)
- **入力**: `raw.properties` (BigQuery) に物件マスタが入っている前提。行動ログは `mlops.search_logs` / `mlops.feedback_events` を Pub/Sub 経由で流し込む
- **担当範囲**: 特徴量加工 → 埋め込み生成 → 候補抽出 + 再ランク → サービング → 監視 → 再学習
- **開発環境**: WSL
- **本番環境**: GCP (`mlops-dev-a` / `asia-northeast1`)
- **言語**: Python 3.12
- **API**: FastAPI + Pydantic
- **IaC**: Terraform 1.9+
- **シークレット管理**: Doppler (値投入のみ、Terraform は container 管理)
- **CI/CD**: GitHub Actions + Workload Identity Federation

## ① `bq-first` — BigQuery 中心 + Cloud Run のみ

### コンセプト

- BigQuery と Cloud Run で完結させる軽量 MLOps
- **Vertex AI / PostgreSQL / BQML 不採用**
- **Meilisearch サーバ採用 (Cloud Run)、Redis サーバは不採用**
- ML 成果物は GCS に格納し、系譜管理は `mlops.training_runs` テーブル + GCS パス命名規則で代替
- 試作・検証フェーズに最適

### アーキテクチャ

```
raw.properties (upstream ETL)
  └─ Dataform ─> feature_mart.properties_cleaned
                 feature_mart.property_features_daily  (ctr / fav_rate / inquiry_rate)
                                 + assertions (quality / freshness)

  └─ Cloud Run Jobs `embedding-job` (multilingual-e5-base)
       └─ feature_mart.property_embeddings (768d FLOAT64 REPEATED + VECTOR INDEX)

  └─ Cloud Run Jobs `training-job` (LightGBM LambdaRank)
       └─ GCS gs://mlops-dev-a-models/lgbm/{date}/{run_id}/model.txt
       └─ mlops.training_runs (NDCG@10 / MAP / Recall@20)

        └─ Cloud Run Service `search-api` (FastAPI)
          ├─ /search   — encoder (lifespan) → Meilisearch(BM25) + VECTOR_SEARCH → RRF fusion → rerank (Phase 6+)
            │             └─ Pub/Sub "ranking-log"    ─> BQ mlops.ranking_log
            ├─ /feedback └─ Pub/Sub "search-feedback" ─> BQ mlops.feedback_events
            └─ /jobs/check-retrain + /events/retrain
                (Cloud Scheduler 04:00 JST → Eventarc → training-job)

        └─ Cloud Run Service `meili-search`
          └─ GCS FUSE mount `/meili_data` (index 永続化)

  └─ Scheduled Query 05:00 JST `property_feature_skew_check`
       └─ mlops.validation_results
            └─ Cloud Monitoring ログベースメトリクス + Looker Studio
```

### 特徴量加工層

- **ツール**: Dataform
- **方式**: SQL + JavaScript、GitHub 連携でバージョン管理
- **スケジュール**: Dataform scheduled executions (日次 03:00 JST)
- **品質保証**: `assertions` で NULL 率・範囲 (ctr / fav_rate / inquiry_rate) チェック + freshness チェック
- **BQ テーブル設計**: partition (`event_date`) + clustering (`property_id`) 必須
- **Training-Serving Skew 対策**: 学習側 SQL (`property_features_daily.sqlx`) と推論側 Python (`build_ranker_features`) を 5 ファイル同一 PR 原則で lockstep 維持

### 埋め込み層

- **モデル**: `intfloat/multilingual-e5-base` (768d、fp32 約 1.1GB)
- **実行基盤**: Cloud Run Jobs `embedding-job` (物件側バッチ) + Cloud Run Service `search-api` (クエリ側オンライン)
- **プロンプト規約**: `"passage: ..."` / `"query: ..."` の prefix を必ず付ける (ME5 の正しい利用法)
- **モデル配置**: GCS `gs://mlops-dev-a-models/encoders/multilingual-e5-base/{version}/` → lifespan で download → sentence-transformers ロード
- **差分更新**: `property_embeddings.text_hash` (sha256(title+description)) が変わった行だけ再エンコード

### 候補抽出層

- **Stage 1a lexical**: Meilisearch (BM25) で top-200 抽出
- **Stage 1b semantic**: BigQuery VECTOR_SEARCH + 属性フィルタで top-200 抽出
- **Stage 2 fusion**: Python 側 RRF (k=60) で rank list を融合し top-100 を確定
- 物件側フィルタ: 家賃上限 (`max_rent`) / 間取り (`layout`) / 徒歩分 (`max_walk_min`) / ペット可 (`pet_ok`) / 築年数 (`max_age`)
- VECTOR INDEX: `CREATE VECTOR INDEX ... OPTIONS(index_type='IVF', distance_type='COSINE')` (Terraform 未対応のため手動 DDL; `docs/04_運用.md STEP 16`)

### 再ランク層

- **モデル**: LightGBM `lambdarank` (group = `request_id`)
- **特徴量** (10 次元): `rent` / `walk_min` / `age_years` / `area_m2` (物件基本) + `ctr` / `fav_rate` / `inquiry_rate` (行動統計 28 日) + `me5_score` / `lexical_rank` / `semantic_rank` (クエリ時シグナル)
- **ラベル Gain**: `inquiry=3 / favorite=2 / click=1 / none=0` (強いアクションが勝つ)
- **評価指標**: NDCG@10 (主)、MAP、Recall@20
- **ハイパラ**: `num_leaves=31 / learning_rate=0.05 / min_data_in_leaf=50 / lambdarank_truncation_level=20 / ndcg_eval_at=[5,10,20]`
- **Phase 4 rerank-free MVP**: booster を lifespan にロードしない状態でも `/search` は候補抽出結果 (`final_rank = lexical_rank`) を返せる。Phase 6 で rerank を bolt-on

### サービング層

- **ランタイム**: Cloud Run Service (Gen2) `search-api`
- **フレームワーク**: FastAPI + Uvicorn
- **モデル配置**: コンテナ同梱せず、lifespan で GCS から encoder + booster をロード
- **Cloud Run 設定**:
  - `--cpu 2 --memory 4Gi` (ME5 encoder + Booster 同居用に 4Gi)
  - `--concurrency 80`
  - `--min-instances 1` (コールドスタート回避)
  - `--max-instances 10`
  - `--cpu-boost`
  - `--execution-environment gen2`
  - `--no-allow-unauthenticated`
- **Dockerfile**: マルチステージ + `uv` パッケージマネージャ
- **認証**: IAM ベース、サービスアカウント分離
- **エンドポイント**:
  - `POST /search` — クエリ埋め込み → Meilisearch(BM25) + VECTOR_SEARCH → RRF fusion → (Phase 6+) rerank → 上位 N 件
  - `POST /feedback` — click / favorite / inquiry イベント publish
  - `POST /jobs/check-retrain` — Cloud Scheduler が 04:00 JST に叩く
  - `POST /events/retrain` — Eventarc 経由で training-job を起動
  - `GET /healthz` `/readyz`

### 監視層

- **ランキング / フィードバックログ**: Pub/Sub → BigQuery Subscription (コード不要)
- **スキュー / ドリフト検出**: カスタム mean-drift 実装 (`mean_drift_sigma`) を Scheduled Query で日次実行
- **可視化**: Looker Studio (無料版)
- **アラート**: Cloud Monitoring のログベースメトリクス (5xx レート / p95 レイテンシ)
- **精度追跡**: `mlops.ranking_log` + `mlops.feedback_events` を join してオフライン NDCG を再計算

### 再学習トリガー層

- **実行環境**: Cloud Run Service 上の `/jobs/check-retrain` エンドポイント
- **スケジュール**: Cloud Scheduler (日次 04:00 JST)
- **トリガー条件** (OR):
  1. `feedback_events` の新規行 > `NEW_FEEDBACK_ROWS_THRESHOLD=10_000` (直近 7d)
  2. 精度低下: 現在の NDCG@10 が 1 週間前から `NDCG_DEGRADATION=0.03` 絶対値で低下
  3. 定期: `STALE_DAYS=7` 日以上学習が走っていない (セーフティネット)
- **キック方式**: Pub/Sub `retrain-trigger` → Eventarc → `/events/retrain` → Cloud Run Jobs `training-job`

### CI/CD

- **トリガー**: GitHub push to main
- **チェック**: `ruff` + `ruff format --check` + `mypy strict` + `pytest` (並列実行)
- **ビルド**: Docker → Artifact Registry (api image / training job image / embedding job image)
- **デプロイ**:
  - `app/**` / `common/**` 変更 → `gcloud run deploy search-api`
  - `jobs/**` / `common/**` 変更 → `gcloud run jobs update training-job`
  - `common/src/common/embeddings/**` 他 → `gcloud run jobs update embedding-job`
  - `definitions/**` 変更 → Dataform API へ compilationResults POST
  - `infra/**` 変更 → `terraform apply` (main マージ時)
- **認証**: すべて WIF (SA Key 禁止)
- **シークレット注入**: Doppler → Cloud Run 環境変数 (未配線、当面は `--set-env-vars` 直書き)

### 不採用 / 条件付き採用

- ❌ **Vertex AI (Pipelines / Training / Endpoints / Model Registry / Feature Store / Experiments / Model Monitoring)** — 軽量方針
- ❌ **BQML** — LambdaRank 非対応、そもそも Vertex AI を呼ぶ形になる
- ❌ **Vertex AI Text Embeddings API** — Vertex AI 不採用方針と整合。ME5 を自前で Cloud Run にロード
- ⚠️ **Meilisearch** — 検索サーバとしては採用 (Cloud Run `meili-search`)。PostgreSQL/Redis 同梱構成は採用しない
- ❌ **Redis / Memorystore** — 現時点では不採用。`cachetools.TTLCache` の in-memory キャッシュを採用
- ❌ **PostgreSQL / Cloud SQL** — すべて BigQuery に寄せる
