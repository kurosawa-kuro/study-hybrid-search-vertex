# GCP-ML Vertex 編 デモシナリオ

**目的:** Phase 3 → Phase 4 の差分を、コードと Vertex コンポーネントの両面から短く見せる
**想定尺:** 7〜10 分
**画面:** ターミナル + エディタ (実 GCP 投入は未実施の前提で、code walk + spec 出力中心)

---

## デモ全体の流れ

1. Port の追加 (`EncoderClient` / `RerankerClient`) を確認
2. Adapter 実装 (`vertex_prediction.py`) の薄さを確認
3. `api.py` lifespan が Vertex クライアント DI に変わっていることを確認
4. `services/ranking.py` が **変わっていない** ことを確認 (境界維持の検証)
5. `infra/modules/vertex/main.tf` で Feature Group + 2 Endpoints + Cloud Function + Eventarc × 2 を確認
6. CPR サーバ 2 本 (`encoder_server.py` / `reranker_server.py`) を確認
7. KFP パイプライン定義 (`embed_pipeline.py` / `train_pipeline.py`) を確認
8. Cloud Function `pipeline_trigger` で Pipeline を submit する経路を確認
9. `scripts/setup/setup_encoder_endpoint.py` / `upload_encoder_assets.py` の spec 出力
10. 3 経路の CT (Monitoring / Scheduler / PipelineJobSchedule) を `scripts/setup/create_schedule.py` で確認
11. `scripts/ops/promote.py` と 9 SA の分離を確認
12. 締め — 「境界を保って差し替える」という設計指針の再確認

---

## Step 1: 新しい Port を確認

### 開くファイル
- `app/src/app/ports/encoder_client.py`
- `app/src/app/ports/reranker_client.py`

### 見せる項目
- `EncoderClient` の `embed(text: str, kind: Literal["query", "passage"]) -> list[float]`
- `RerankerClient` の `predict(instances: list[list[float]]) -> list[float]`
- **どちらも Protocol のみ、`google.cloud` への import は無い**

### ナレーション要点

> Port は Protocol として純粋に型だけを定義します。
> Vertex SDK への依存はここでは持ちません。
> これは AST 検査が強制しているので、うっかり `from google.cloud import aiplatform` を書き込むと CI が止まります。

## Step 2: Adapter の薄さを確認

### 開くファイル
- `app/src/app/adapters/vertex_prediction.py`

### 注目箇所
- `_normalize_endpoint_name` (endpoint_id を resource name に正規化)
- `_create_endpoint` (`aiplatform.init` + `aiplatform.Endpoint`)
- `VertexEndpointEncoder.embed` — `predict(instances=[{"text": f"{kind}: {text}"}])`
- `VertexEndpointReranker.predict` — `predict(instances=instances)`

### ナレーション要点

> Adapter 側は `aiplatform.Endpoint.predict` を叩く薄い wrapper だけです。
> エラー処理とレスポンスの正規化に寄せていて、ロジックらしいロジックは入っていません。
> Phase 3 で言えば `BigQueryCandidateRetriever` と同じ層にいる、という理解でいいです。

## Step 3: `api.py` lifespan の配線

### 開くファイル
- `app/src/app/entrypoints/api.py`

### 注目箇所
- `_build_encoder_client` / `_build_reranker_client` のファクトリ関数
- `lifespan` 内で `app.state.encoder_client` / `app.state.reranker_client` を set
- 旧 `model_store.py` を import していない点
- `/events/retrain` ハンドラが **存在しない** 点

### ナレーション要点

> lifespan で Vertex クライアントを組み立てて、app.state に載せるだけになりました。
> Phase 3 にあった Booster のローカルロードや、CloudRunJobRunner の配線はもう残っていません。
> この lifespan の短さが Phase 4 の軽さを象徴しています。

## Step 4: `services/ranking.py` が変わっていないことを確認

### 開くファイル
- `app/src/app/services/ranking.py`

### 注目箇所
- `run_search` 関数の骨格 (RRF → rerank → top-K)
- 引数として `reranker: RerankerClient | None` を受ける
- Port 型しか見ておらず、`google.cloud` の import がない

### ナレーション要点

> services 側は、Phase 3 からほとんど変わっていないことを確認します。
> Port 越しにしか rerank を呼ばない設計のおかげで、adapter が Vertex に差し替わっても、本体のロジックは書き換え不要でした。
> Port/Adapter 境界の効果がここで出ています。

## Step 5: `infra/modules/vertex/main.tf` ツアー

### 開くファイル
- `infra/modules/vertex/main.tf`

### 注目箇所
- `google_vertex_ai_feature_group.property_features` + `_feature` × 7 (`feature_group_property_features` local)
- `google_vertex_ai_endpoint.encoder` / `.reranker` — shell だけ宣言、`deployed_models` / `traffic_split` は書かない
- `google_pubsub_topic.model_monitoring_alerts` + BQ Subscription
- `google_cloudfunctions2_function.pipeline_trigger` + `functions/pipeline_trigger/` の zip upload
- `google_eventarc_trigger.retrain_to_pipeline` / `.monitoring_to_pipeline`
- `google_storage_bucket_iam_member.endpoint_{encoder,reranker}_models_reader`

### ナレーション要点

> Vertex 用に新設したモジュールで、6 モジュール目になります。
> Feature Group は Dataform テーブルを wrap するだけの薄い宣言です。
> Endpoint は空の shell だけで、Model deploy は Python SDK 側の責務です。
> Cloud Function と Eventarc 二つが、Phase 3 の Cloud Run Job 経路を置き換えています。

## Step 6: CPR サーバ 2 本

### 開くファイル
- `jobs/src/training/entrypoints/encoder_server.py`
- `jobs/src/training/entrypoints/reranker_server.py`

### 注目箇所
- `AIP_HEALTH_ROUTE` / `AIP_PREDICT_ROUTE` / `AIP_HTTP_PORT` を env から読む
- encoder は `common.embeddings.e5_encoder.E5Encoder.load` をそのまま使う
- reranker は `lgb.Booster(model_file=...)` をそのまま使う
- どちらも `main()` で `uvicorn.run(app, ...)`、`python -m` で起動できる
- `jobs/containers/{encoder,reranker}/Dockerfile` が呼び出す

### ナレーション要点

> Vertex の CPR 規約に従った FastAPI サーバを二本書きます。
> 重要なのは、推論ロジック本体を Phase 3 から import してそのまま使っている点です。
> encoder は E5Encoder のクラスを、reranker は LightGBM の booster 読み込みをそのまま使います。
> 規約対応は薄い入れ物を書くだけで済みます。

## Step 7: KFP パイプライン定義

### 開くファイル
- `pipelines/src/pipelines/property_search/embed_pipeline.py`
- `pipelines/src/pipelines/property_search/train_pipeline.py`
- `pipelines/src/pipelines/property_search/components/` 配下

### 注目箇所
- `embed_pipeline`: `load_properties → batch_predict_embeddings → write_embeddings` の 3 段
- `train_pipeline`: `load_features → (vizier 分岐) → train_reranker → evaluate → register_reranker`
- `train_reranker` は container_component で `rank_trainer.py::train` を呼ぶ
- `register_reranker` は `ModelUploadOp` + `Model.deploy` でカナリア

### 例コマンド

```bash
uv run python -m pipelines.property_search.compile --target embed
uv run python -m pipelines.property_search.compile --target train
```

### ナレーション要点

> KFP は DAG を YAML にコンパイルして、GCS に置き、PipelineJob で submit する構成です。
> train パイプラインは Phase 3 の rank_trainer の train 関数をそのまま container_component で包んでいるのがわかります。
> コンパイルだけならここで見せて、実 submit は GCP 側の話として締めます。

## Step 8: Cloud Function `pipeline_trigger`

### 開くファイル
- `functions/pipeline_trigger/main.py`
- `infra/modules/vertex/main.tf` の `google_cloudfunctions2_function.pipeline_trigger` ブロック

### 注目箇所
- `trigger_pipeline(cloud_event)` が entry
- env (`PROJECT_ID` / `VERTEX_LOCATION` / `PIPELINE_TEMPLATE_URI` / `PIPELINE_ROOT` / `PIPELINE_SERVICE_ACCOUNT`) は Terraform が注入
- `aiplatform.PipelineJob(template_path=..., parameter_values=...).submit(service_account=sa-pipeline)` で submit
- event source (retrain-trigger / monitoring alerts) で `enable_tuning` の default を切替

### ナレーション要点

> Phase 3 で Eventarc の subscriber だった Cloud Run Job が、Cloud Function に差し替わっています。
> Cloud Function は、受け取った CloudEvent を元にどちらの経路から来たかを判別し、PipelineJob を submit します。
> Monitoring アラート由来のときだけ Vizier を ON にするのが、コストと品質のバランスを取る工夫です。

## Step 9: encoder 初回投入の one-off スクリプト

### 開くファイル
- `scripts/setup/upload_encoder_assets.py`
- `scripts/setup/setup_encoder_endpoint.py`

### 例コマンド (spec 出力のみ、apply しない)

```bash

uv run python -m scripts.setup.setup_encoder_endpoint
```

### 画面で見るポイント
- `build_upload_spec` の出力 (`model_name` / `bucket` / `prefix` / `gcs_uri`)
- `build_endpoint_spec` の出力 (`serving_container_image_uri` / `artifact_uri` / `machine_type` / `service_account` / `traffic_percentage`)

### ナレーション要点

> reranker は train パイプラインの register_reranker コンポーネントが自動でやりますが、encoder は別パスです。
> ここでは spec だけを出して、実際の `--apply` は GCP 側で叩く、という扱いにします。

## Step 10: 3 経路の CT と Schedule

### 開くファイル
- `scripts/setup/create_schedule.py`
- `scripts/setup/setup_model_monitoring.py`

### 画面で見るポイント

> retrain_policy.py はそのまま流用、Cloud Scheduler → check-retrain → Pub/Sub publish の導線は Phase 3 と同一です。

> 9 SA の最小権限設計は、Endpoint 実行権限と Pipeline 起動権限を厳密に分ける狙いです。

## Step 12: 締め

### 伝えること
- Phase 3 → 4 は **差分工事**、検索コアと特徴量と学習ロジックは温存
- Vertex は 7 コンポーネントが工程を分担、Port/Adapter 境界で着脱可能に保つ
- Cloud Run Jobs は KFP に、Eventarc subscriber は Cloud Function に移行済
- Dual write (Registry / training_runs、Experiments / W&B) で既存運用導線を壊さない

### クロージング例

> この教材では、Phase 3 の bq-first に Vertex AI レイヤをどう後付けしたかを見てきました。
> コードを追うときは、「変わった場所」と同じくらい「変わっていない場所」を意識してください。
> 境界を守って差し替えるとはどういうことか、vertex_prediction.py と ranking.py を見比べると一番わかりやすいです。
