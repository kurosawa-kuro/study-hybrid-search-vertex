# GCP-ML デモシナリオ

**目的:** MLOps パイプライン全体と、GCP 技術がどうつながるかを、コードと運用コマンドで短く見せる  
**想定尺:** 5〜7 分  
**画面:** ターミナル + エディタ + 必要に応じて BigQuery テーブル確認

---

## デモ全体の流れ

1. `SearchRequest` を確認して API の入力を把握する
2. `ranking.py` を開いて `/search` の主処理を読む
3. `lexical_search.py` を開いて Meilisearch `BM25` を確認する
4. `candidate_retriever.py` を開いて `BigQuery VECTOR_SEARCH` と `RRF` を確認する
5. `cache_store.py` を開いて in-memory `TTLCache` を確認する
6. `make ops-search` を実行して検索結果を見る
7. `make ops-ranking` で順位情報を見る
8. `training_runs` / `ranking_log` / `feedback_events` の位置付けに触れる
9. `make ops-check-retrain` で再学習導線を説明する
10. 各工程と GCP 技術の対応を締めで整理する

---

## Step 1: API 入力の確認

### 開くファイル
- `app/src/app/schemas/search.py`

### 見せる項目
- `query`
- `limit`
- `max_rent`
- `layout`
- `max_walk_min`
- `pet_ok`

### ナレーション要点

> この API は推論工程の入口です。  
> ここで受けた条件が、Meilisearch と BigQuery の候補抽出、そして後段の rerank につながります。

---

## Step 2: `/search` 主処理の確認

### 開くファイル
- `app/src/app/services/ranking.py`

### 注目箇所
- `run_search`
- query embedding
- candidate retrieval
- `_score_candidates`
- ranking log publish

### ナレーション要点

> 主線はここです。  
> クエリ埋め込みを作り、lexical と semantic の二系統検索をまとめ、必要なら booster で並べ替え、最後に ranking_log を publish します。  
> API の責務が、HTTP 受付だけでなく、推論工程全体のオーケストレーションだと分かります。

---

## Step 3: Meilisearch BM25 の確認

### 開くファイル
- `app/src/app/adapters/lexical_search.py`

### 見せる箇所
- `MeilisearchLexical`
- `/indexes/properties/search`
- filter 組み立て
- `property_id, lexical_rank`

### ナレーション要点

> lexical 側は Meilisearch です。  
> Phase 2 Local と同じ BM25 の考え方を維持したまま、Phase 3 では Cloud Run Service として切り出しています。  
> 語句一致が強い候補をまず別系統で拾うことが、semantic 検索だけに寄せない理由です。

---

## Step 4: BigQuery VECTOR_SEARCH と RRF の確認

### 開くファイル
- `app/src/app/adapters/candidate_retriever.py`

### 見せる箇所
- `BigQueryCandidateRetriever`
- `_semantic_search`
- SQL の WHERE 条件
- `VECTOR_SEARCH`
- `me5_score`
- `rrf_fuse`
- `lexical_rank`, `semantic_rank`, `rrf_rank`

### ナレーション要点

> 候補抽出は一段ではありません。  
> semantic 側は BigQuery VECTOR_SEARCH が担当し、lexical 側の結果と RRF で融合します。  
> この教材では、BigQuery が保存先ではなく、semantic 検索と特徴量参照を支える点が重要です。

---

## Step 5: in-memory TTL cache の確認

### 開くファイル
- `app/src/app/adapters/cache_store.py`

### 見せる箇所
- `InMemoryTTLCacheStore`
- `cachetools.TTLCache`
- `default_ttl_seconds=120`

### ナレーション要点

> キャッシュは Redis ではなく、process-local の in-memory TTL cache です。  
> Cloud Run 上の search-api に閉じた軽量構成で、TTL は 120 秒です。  
> ここも Phase 2 の Redis から、Phase 3 で運用を簡素化したポイントです。

---

## Step 6: 検索を実行する

### 例コマンド

```bash
make ops-search
```

### 画面で見るポイント
- 応答が返ること
- `me5_score`
- `semantic_rank`
- `rrf_rank`
- `final_rank`
- `score`

### ナレーション要点

> ここでは、検索結果が実際に返ることを確認します。  
> booster が載っていれば score が入り、未配置なら lexical rank ベースで返る設計です。  
> どちらの場合でも API 自体は成立する、という段階設計が見どころです。

---

## Step 7: 順位情報の確認

### 例コマンド

```bash
make ops-ranking
```

### 画面で見るポイント
- `lexical_rank`
- `semantic_rank`
- `rrf_rank`
- `final_rank`
- `score`
- `me5_score`

### ナレーション要点

> ここでは、lexical、semantic、RRF、最終順位の違いに注目します。  
> `me5_score` は特徴量の一つであって、そのまま最終順位ではありません。  
> ML がどこで介入しているかを、この画面で説明します。

---

## Step 8: BigQuery に何が残るか

### 触れるテーブル
- `feature_mart.property_embeddings`
- `mlops.training_runs`
- `mlops.ranking_log`
- `mlops.feedback_events`

### ナレーション要点

> このシステムは、検索して終わりではありません。  
> 物件埋め込み、学習履歴、返却順位、ユーザー反応が残るので、推論と再学習の間がつながっています。  
> ここが MLOps パイプラインのループ部分です。

---

## Step 9: 再学習導線を確認する

### 例コマンド

```bash
make ops-check-retrain
```

### 画面で見るポイント
- `reasons`
- `published`
- 条件判定の結果

### ナレーション要点

> 再学習は毎回無条件で回すのではなく、条件判定を通します。  
> フィードバック増加、NDCG 悪化、一定期間未学習のいずれかで trigger されます。  
> ここで、運用まで含めて GCP サービスがつながっていることを示します。

---

## Step 10: 締め

### 伝えること
- Cloud Run は API と Job の両方に使っている
- BigQuery は保存、特徴量、semantic 検索、ログを横断して使っている
- Pub/Sub, Scheduler, Eventarc が再学習ループをつないでいる

### クロージング例

> この教材では、MLOps パイプライン全体と、GCP 技術がどう工程分担しているかを見ました。  
> 実装を読むときは、API だけでなく、データ加工、ログ、再学習導線まで一緒に追うのがポイントです。
