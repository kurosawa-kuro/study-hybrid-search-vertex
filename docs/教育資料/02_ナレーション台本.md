**前提:** Phase 1 / 2 / 3 の教材を受講済。Meilisearch BM25 + BigQuery VECTOR_SEARCH + RRF + LightGBM LambdaRank の構成は既知
**方針:** Phase 3 → 4 の差分と、Vertex AI プリミティブの役割分担に集中

## [Slide 00] タイトル - 15 秒

> GCP-ML Vertex 編です。
> 今回は、Phase 3 の bq-first パイプラインに Vertex AI レイヤを後付けしたプロジェクトを読み解いていきます。
> 差分工事であって、ゼロから作り直すわけではないのがポイントです。

## [Slide 01] 前提とゴール - 30 秒

> Phase 1 の ML 基礎、Phase 2 の Local ハイブリッド検索、そして Phase 3 の GCP 移植は受講済という前提で進めます。
> ゴールは三つです。
> 一つ目、Phase 3 から Phase 4 で何を置き換え、何を残すのかを言えること。
> 二つ目、Vertex AI プリミティブがどの工程を担当するかを整理できること。
> 三つ目、Port と Adapter の境界を保って adapter だけ差し替えた意味を説明できることです。

## [Slide 02] この教材の位置付け - 25 秒

> 題材は Phase 3 から不変で、不動産検索の三段構成を使います。
> 強調したいのは、検索コアや特徴量や学習ロジックをそのまま残し、境界の外側だけを Vertex に差し替える工事の進め方です。
> 採用する Vertex 機能と、あえて採用しない機能の trade-off も同じ重みで扱います。

## [Slide 03] フェーズ別 stack 整理 - 50 秒

> 左が Phase 3、右が Phase 4 の対応表です。
> 学習オーケストレーションは Cloud Run Jobs から Vertex AI Pipelines の KFP 二本に置き換わりました。
> rerank 推論とクエリ埋め込みは、lifespan で同居ロードする形から Vertex Endpoint 二本へ外部化されます。
> 物件バッチ埋め込みは embed パイプライン側の Batch Prediction ステップになります。
> Model 管理は Vertex AI Model Registry、特徴量は Vertex Feature Group で Dataform テーブルをラップします。
> 監視は Model Monitoring v2 が正系に変わり、旧 SQL はバックアップとして残します。
> Cloud Run Service のメモリは 4 ギガから 2 ギガに落とせて、サービスアカウントは 5 個から 9 個に増えています。

## [Slide 04] 採否が変わらない項目 - 40 秒

> 意図的に Phase 3 を継続する項目も多いです。
> Meilisearch の BM25 は Vertex に相当物が無いので Cloud Run 上にそのまま残します。
> BigQuery の VECTOR_SEARCH も、遅延が十分出ていて IaC の手動 DDL も定着しているため据え置きです。
> Dataform、Pub/Sub から BigQuery へのサブスクリプション、5 SA 分離と Workload Identity Federation は Phase 3 から不変です。
> 監視 SQL も、Vertex Monitoring が落ちたときのフォールバックとして残します。
> Port と Adapter の境界と AST 検査は、新しい adapter を足すだけで効くので壊す必要がありません。

## [Slide 05] Vertex コンポーネント対応表 - 50 秒

> Vertex 側で使うのは七つのコンポーネントです。
> Pipelines が学習と埋め込みの DAG、Google Cloud Pipeline Components が BigQuery クエリやバッチ予測、モデル登録の標準オペレータを提供します。
> Feature Group は Dataform テーブルの上にかぶせるオフラインの wrapper で、lineage を Vertex 側に残すのが目的です。
> Model Registry は staging と production の二段エイリアスで、promote スクリプトで切り替えます。
> Endpoints は encoder と reranker の二本、Experiments は W&B と dual write、Vizier はハイパラ探索、Monitoring は日次ドリフト検知です。
> 一部の機能、たとえば ML Metadata や Pipeline Templates は KFP 実行と CI で自動生成されます。

## [Slide 06] search-api がプロキシになる - 45 秒

> Phase 3 の search-api は、lifespan で multilingual-e5 エンコーダと LightGBM booster を同居ロードしていたので、メモリは 4 ギガ必要でした。
> Phase 4 の search-api は、Vertex Endpoint のクライアントを DI するだけになります。
> torch と LightGBM の依存を API 側から外せるので、メモリは 2 ギガに落ちます。
> 重要なのは、ranking サービスの run_search 関数、つまり RRF、rerank、top-K の並び替えロジックは変えない点です。
> プロキシ化しても、オーケストレーションの責務はそのまま API に残しています。

## [Slide 07] Port/Adapter 境界の維持 - 40 秒

> 追加した Port は二つです。
> EncoderClient と RerankerClient、どちらも Protocol 定義なので Vertex に依存しません。
> 実装は app の adapters 配下の vertex prediction ファイルにまとめ、aiplatform Endpoint predict を叩く薄い wrapper だけです。
> services の ranking が Port 型しか見ないので、ユニットテストは stub adapter を刺すだけで済みます。
> さらに AST 検査が、Port 側に google.cloud の import が紛れ込まないことを自動で確認してくれます。

## [Slide 08] CPR カスタムコンテナ 2 種 - 45 秒

> Vertex Endpoint の裏に載せる FastAPI サーバを二本書きます。
> どちらも Vertex の Custom Prediction Routine 規約、つまり AIP_HEALTH_ROUTE と AIP_PREDICT_ROUTE と AIP_HTTP_PORT に従います。
> encoder 側は、起動時に AIP_STORAGE_URI から ME5 チェックポイントをダウンロードし、common.embeddings の E5Encoder をそのまま import して推論を回します。
> reranker 側も同じ規約で、model.txt を落として LightGBM booster をロードするだけです。
> Phase 3 の学習ロジックや推論ロジックを書き換えないで済むのが、この CPR コンテナ化のうれしさです。

## [Slide 09] KFP パイプライン 2 本 - 45 秒

> 一本目の embed パイプラインは、load_properties、batch_predict_embeddings、write_embeddings の三段です。
> Phase 3 の embedding-job にあった text_hash 差分ロジックをそのまま載せ、encoder Endpoint を Batch Prediction で叩きます。
> 二本目の train パイプラインは、load_features から train_reranker、evaluate、条件分岐を経て register_reranker まで進みます。
> 学習の本体は、既存の rank_trainer の train 関数を container_component で包むだけで、LightGBM ハイパラや NDCG 評価は Phase 3 から不変です。

## [Slide 10] Model Registry とカナリアデプロイ - 40 秒

> Registry には ModelUploadOp で version を足し、エイリアスを staging で初期化します。
> Endpoint へのデプロイは、新モデル 10 パーセント、旧モデル 90 パーセントのカナリアから始めます。
> 手動昇格は scripts ops の promote スクリプトで、production エイリアスを付け替え、トラフィックを 100 パーセントに切り替え、旧モデルは undeploy してコストを削ります。
> Registry と BigQuery の training_runs テーブルは dual write で併存させて、SQL からのアドホック集計を残します。

## [Slide 11] Experiments の二重書き込み - 35 秒

> Vertex Experiments は、Registry、Pipelines、ML Metadata と自動で lineage が繋がるのが強みです。
> 一方、W&B は Phase 1 から使っている可視化とアドホック比較に強く、社外でも定着しています。
> この二つを両方書き込む構成にしていて、VERTEX_EXPERIMENT_NAME という環境変数が立っている時だけ Vertex 側にも追記します。
> Phase 3 の BigQuery INSERT は消さずに残すので、既存の SQL 集計は何も変わりません。

## [Slide 12] Feature Group と 6 ファイル parity - 45 秒

> Phase 3 には、特徴量の整合性を守るための五ファイル lockstep がありました。
> feature_schema、feature_engineering、infra の ranking_log RECORD、monitoring の SQL、Dataform の SQLX、の五本です。
> Phase 4 ではここに六本目として、Vertex Feature Group のリソース宣言が加わります。
> property 側の七列について、Terraform の main.tf で feature 名と DOUBLE 型を宣言し、パリティテストが FEATURE_COLS_RANKER と一対一対応であることを AST で検査します。
> クエリ時算出の me5_score、lexical_rank、semantic_rank は Feature Group には入れないのも、Phase 3 の監視 SQL と同じ扱いです。

## [Slide 13] Continuous Training の 3 経路 - 50 秒

> 再学習の起動経路は三つあります。
> 一つ目は Cloud Scheduler が日次で check-retrain を叩き、条件成立で retrain-trigger トピックに publish する既存経路です。
> 二つ目は Vertex Model Monitoring v2 がドリフトを検知して、model-monitoring-alerts トピックに流す経路です。
> 三つ目は PipelineJobSchedule がセーフティネットとして、embed パイプラインは日次、train パイプラインは日曜だけ起動します。
> 一つ目と二つ目は Eventarc 経由で Cloud Function に届き、そこから aiplatform の PipelineJob submit を呼びます。
> Phase 3 で Cloud Run Job を叩いていた subscriber が、Phase 4 では Cloud Function に切り替わっているのが大きな差です。

## [Slide 14] 9 サービスアカウント分離 - 40 秒

> Phase 3 は 5 SA + WIF でした。
> Phase 4 では、sa-pipeline、sa-endpoint-encoder、sa-endpoint-reranker、sa-pipeline-trigger の四つを追加します。
> さらに sa-api には、Endpoint を呼ぶために aiplatform.user を足します。
> Endpoint のランタイム SA と、Pipeline を起動する SA を分けておくのは、ドリフト時の blast radius を小さく保つためです。
> sa-job-embed だけは、embed パイプラインの定着後に段階的に削除する予定で、いまは継続しています。

## [Slide 15] Monitoring v2 とカナリア - 40 秒

> Monitoring のセットアップは、scripts setup の setup_model_monitoring が spec を組み立てるところまでを担当します。
> 学習データセットは feature_mart の property_features_daily で、属性は FeatureDriftSpec と PredictionOutputDriftSpec の二種類です。
> アラートは Pub/Sub の model-monitoring-alerts トピックから BigQuery の model_monitoring_alerts テーブルに流れ、同時に Eventarc で Cloud Function にも届きます。
> ドリフトが発火したときだけ、Cloud Function 側で enable_tuning を真にして train パイプラインを submit し、Vizier によるハイパラ探索を混ぜます。

## [Slide 16] 残す判断とやめる判断 - 40 秒

> 残すと決めたものは三つあります。
> Meilisearch と BigQuery の VECTOR_SEARCH と RRF、Dataform、そして監視 SQL です。
> Vertex に相当物がない、あるいは Matching Engine 化が別スコープ、という理由で据え置いています。
> やめたものも三つです。
> Cloud Run Jobs の training-job と embedding-job、Phase 3 の booster local ロード経路、そして /events/retrain と CloudRunJobRunner です。
> いずれも Vertex プリミティブに役割が移ったため dead code になり、削除しました。

## [Slide 17] ハマりどころ - 45 秒

> 実作業で落ちやすい点を三つ共有します。
> 一つ目、Vertex Endpoint の deployed_models と traffic_split は provider 側で computed 扱いなので、Terraform 側には宣言できず、ignore_changes も書けません。deploy は Python SDK 側でやります。
> 二つ目、encoder の初回投入は自動化されていません。reranker は train パイプラインが自動でやりますが、encoder だけは upload_encoder_assets と setup_encoder_endpoint の二つのスクリプトを手で叩きます。
> 三つ目、VECTOR INDEX は Terraform provider が未対応なので、CREATE VECTOR INDEX の DDL は手動です。Doppler の Secret Manager 経路は意図的に配線していません。

## [Slide 18] 全体アーキテクチャ - 45 秒

> 全体像をまとめます。
> 左上の raw properties から Dataform 経由で feature_mart が作られ、その上に Feature Group がかぶります。
> embed パイプラインが encoder Endpoint を叩いて property_embeddings を更新します。
> 中央の search-api は、encoder Endpoint、Meilisearch、BigQuery VECTOR_SEARCH、reranker Endpoint をオーケストレーションするプロキシです。
> 右側の train パイプラインは、load_features から Vizier、train_reranker、evaluate、register_reranker と進みます。
> 下では Monitoring、Scheduler、Schedule の三経路が Cloud Function に集約され、PipelineJob submit が走ります。

## [Slide 19] デモで見るべき観点 - 40 秒

> デモでは、コードと Vertex コンポーネントの両方を追ってください。
> コード側では、api.py の lifespan、vertex_prediction の adapter、ranking サービスが変わっていないこと、それから infra modules vertex の Terraform を確認します。
> Vertex コンポーネント側では、embed と train の KFP DAG、CPR サーバ 2 本、pipeline_trigger の Cloud Function、setup_encoder_endpoint の spec 出力が中心です。
> 「どこが変わった」と同じくらい「どこが変わっていない」を確認するのがこの教材の主眼です。

## [Slide 20] この教材で押さえるべきこと - 40 秒

> 五点まとめます。
> 一点目、Phase 3 の検索コアと特徴量と学習ロジックはそのまま、adapter だけを Vertex 実装に差し替えます。
> 二点目、Vertex は 7 コンポーネントが工程を分担します。
> 三点目、Cloud Run Jobs が KFP に移り、再学習 subscriber が Cloud Function に移りました。
> 四点目、Registry と training_runs、Experiments と W&B の dual write で運用導線を壊しません。
> 五点目、CPR コンテナ 2 本と Endpoint 2 本が、Phase 4 で新しく学ぶ実装対象です。

## [Slide 21] まとめ - 30 秒

> このリポジトリは、Phase 3 の bq-first に Vertex AI レイヤを後付けした差分プロジェクトです。
> 境界を守って差し替える、というのが一貫した設計指針で、Port と Adapter と parity テストがその境界を支えています。
> 次の一歩として、vertex_prediction の adapter の薄さを読み、infra modules vertex を眺め、setup_encoder_endpoint の spec を出してみてください。
