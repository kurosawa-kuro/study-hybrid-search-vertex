## [Slide 00] タイトル - 10 秒

> GCP-ML 入門です。  
> 今回は、Cloud Run と BigQuery を中心にした MLOps パイプラインの学習素材として、このリポジトリの実装を見ていきます。

## [Slide 01] 前提とゴール - 25 秒

> すでに機械学習の基礎は受講済み、という前提で進めます。  
> そのため今回は、特徴量や推論の復習よりも、GCP 上でどう構成しているかに集中します。  
> ゴールは三つです。  
> 一つ目、MLOps パイプラインの工程を説明できるようになること。  
> 二つ目、GCP の各技術がどこで関与するか整理できるようになること。  
> 三つ目、再学習まで含めた運用ループを説明できるようになることです。

## [Slide 02] この教材の位置付け - 25 秒

> 題材は不動産検索ですが、主役はドメインではありません。  
> GCP 上で、データ加工、学習、推論、ログ収集、再学習がどうつながるかを学ぶことが目的です。  
> Vertex AI を使わずに、軽量な MLOps をどう組むかを見る教材だと捉えてください。

## [Slide 03] フェーズ別技術スタック整理 - 45 秒

> この教材は三段階の流れの三つ目に当たります。  
> Phase 1 は住宅価格予測で LightGBM 回帰を学ぶ基礎編です。  
> Phase 2 は Local 環境で、Meilisearch の BM25 と multilingual-e5 と LightGBM reranker を組み合わせた検証編です。  
> そして Phase 3 が今回の対象で、Meilisearch を Cloud Run に載せ、semantic 検索を BigQuery VECTOR_SEARCH に置き換え、RRF と LambdaRank まで含めた実務寄り構成です。  
> つまり検索思想は継承しつつ、実行基盤を GCP 向けに再構成したフェーズだと捉えてください。

## [Slide 04] MLOps パイプラインの全体像 - 35 秒

> まず工程を並べます。  
> 生データを受け、特徴量を作り、埋め込みを作り、学習し、API で推論し、ログを集め、条件に応じて再学習します。  
> この一周が MLOps パイプラインです。  
> 今回は、この各工程を GCP のどの技術が担当するかを見ていきます。

## [Slide 05] なぜ Cloud Run と BigQuery か - 35 秒

> Cloud Run は、API もバッチ Job も同じ延長で扱えるのが強みです。  
> 一方 BigQuery は、保存先だけでなく、特徴量の置き場、ログ分析、semantic 候補抽出の実行基盤にもなります。  
> さらに lexical 側は Meilisearch を Cloud Run に載せることで、Phase 2 と同じ検索の考え方を維持できます。  
> この組み合わせで、Local の検索設計を GCP へ持ち上げています。

## [Slide 06] GCP 技術と工程の対応 - 40 秒

> 生データとログの置き場は BigQuery です。  
> 特徴量加工は Dataform と BigQuery、埋め込み生成と学習は Cloud Run Jobs、モデル保存は GCS、推論 API は Cloud Run Service が担当します。  
> 検索では Meilisearch が lexical、BigQuery が semantic と特徴量参照を受け持ちます。  
> さらに Pub/Sub と Eventarc と Scheduler が、ログ連携と再学習起動をつなぎます。  
> つまり、各技術は独立しているのではなく、工程ごとに分担して一つのループを作っています。

## [Slide 07] 採用技術の役割分担 - 35 秒

> API は FastAPI を Cloud Run に載せています。  
> lexical 検索は Meilisearch の BM25 を Cloud Run で動かし、semantic 検索は BigQuery VECTOR_SEARCH を使います。  
> 特徴量加工は Dataform、埋め込みは multilingual-e5-base、最終順位は LightGBM LambdaRank です。  
> さらに search-api の手前には cachetools.TTLCache の in-memory キャッシュがあります。  
> さらに GCS がモデル置き場、Pub/Sub と Eventarc と Scheduler が再学習のトリガー担当です。  
> つまり、GCP サービス群の組み合わせで MLOps パイプラインを構成しています。

## [Slide 08] 全体アーキテクチャ - 40 秒

> まず raw.properties があり、Dataform が feature mart を作ります。  
> 次に embedding-job が物件埋め込みを作って、BigQuery に保存します。  
> そして training-job が ranking_log と feedback_events を使って学習し、モデルを GCS に保存します。  
> search-api は、そのモデルに加えて Meilisearch と BigQuery の二系統検索を使って検索結果を返します。  
> 一つの API の裏に、データ加工と学習のパイプラインがぶら下がっている構図です。

## [Slide 09] データ加工の流れ - 35 秒

> Dataform では、物件マスタをそのまま使うのではなく、cleaned テーブルや daily features テーブルに整えています。  
> ここで CTR や favorite rate、inquiry rate のような行動特徴も作ります。  
> 重要なのは、学習用 SQL と推論側 Python が同じ特徴量定義を前提にしていることです。  
> これが training serving skew を避ける基本です。

## [Slide 10] 埋め込み生成の流れ - 40 秒

> 物件側の埋め込みは embedding-job がバッチで作ります。  
> 物件説明文に passage prefix を付けて埋め込みを生成し、property_embeddings テーブルへ保存します。  
> 一方クエリ側は search-api がオンラインで処理します。  
> query prefix を付けて埋め込みを作り、semantic 側の BigQuery VECTOR_SEARCH に使います。  
> lexical 側の Meilisearch は同じクエリ文字列を BM25 で評価します。  
> 重い物件側は事前計算、軽いクエリ側だけオンライン、という分担です。

## [Slide 11] 推論工程の入口 - 30 秒

> SearchRequest では、検索文だけでなく、家賃上限、間取り、徒歩分、ペット可否などの条件も受け取ります。  
> つまりこの API は、意味検索だけでも、厳密フィルタだけでもなく、その両方を受けられる設計です。  
> ここで受けた条件が、そのまま候補抽出の WHERE 句に効いてきます。

## [Slide 12] 推論工程の処理順 - 45 秒

> 処理の流れは明快です。  
> まず search-api がリクエストを受け、クエリ埋め込みを作ります。  
> 次に Meilisearch の BM25 と BigQuery VECTOR_SEARCH の二系統で候補を取ります。  
> その二つを RRF で融合し、候補集合を一つにまとめます。  
> その候補に対して ranker 用特徴量を組み立て、LightGBM があれば再ランキングします。  
> 最後に ranking_log を残してレスポンスを返します。  
> この順番を頭に入れると、コードが追いやすくなります。

## [Slide 13] 候補抽出は 2 系統で行う - 40 秒

> この教材で重要なのは、候補抽出が一系統ではない点です。  
> lexical は Meilisearch の BM25、semantic は BigQuery VECTOR_SEARCH が担当します。  
> その二つを RRF で合わせることで、語句一致の強さと意味の近さを両立します。  
> さらに統合後の特徴量参照とログ分析は BigQuery に寄せているので、検索と分析が分断しません。

## [Slide 14] 再ランキングの役割 - 35 秒

> 候補が出たあと、最終順位は LightGBM LambdaRank が決めます。  
> 入る特徴量は、家賃や徒歩分といった物件属性に加えて、CTR や inquiry rate、さらに me5_score、lexical_rank、semantic_rank、rrf_rank です。  
> つまり、意味の近さだけでなく、検索順位と行動実績も合わせて順位を決めています。  
> これが推論工程の中核です。

## [Slide 15] モデルが無いときも API は動く - 30 秒

> 実装上は、booster がまだ無くても `/search` は返ります。  
> その場合は rerank を省いて lexical rank をそのまま final rank にします。  
> この段階分けによって、候補抽出の疎通確認と、rerank 導入後の品質改善を切り分けて学べます。

## [Slide 16] ログとフィードバックの工程 - 35 秒

> API は結果を返して終わりではありません。  
> ranking_log に、どの順番で返したかを残します。  
> ここには lexical_rank、semantic_rank、rrf_rank も残るので、どの段で効いたかを後から分析できます。  
> さらに feedback_events に click や favorite や inquiry を蓄積します。  
> この二つが、次回の学習データになります。  
> つまり、推論の出力がそのまま再学習の入力候補になります。  
> ここが、MLOps パイプラインのループ部分です。

## [Slide 17] 再学習の工程 - 40 秒

> 再学習は、Cloud Scheduler が毎日 check-retrain を叩くところから始まります。  
> 条件は、フィードバック件数の増加、NDCG の悪化、あるいは学習の停滞です。  
> 条件に当たると、Pub/Sub の retrain-trigger を経由して Eventarc が training-job を起動します。  
> オンライン API とバッチ学習が、GCP のイベントで接続されている形です。

## [Slide 18] Cloud Run が担う工程 - 35 秒

> search-api は Service、embedding-job と training-job は Jobs です。  
> さらに lexical 用の Meilisearch も独立 Cloud Run Service として動かしています。  
> どちらも Cloud Run に寄せているので、Docker、IAM、デプロイの考え方をそろえられます。  
> 学習素材としては、GCP 上での API とバッチの共通点を説明しやすい構成です。

## [Slide 19] デモで見るべき観点 - 35 秒

> デモでは、API コードだけでなく、Meilisearch 呼び出しと BigQuery のテーブルもセットで確認してください。  
> `property_embeddings` があること、`training_runs` にモデル履歴が残ること、`ranking_log` と `feedback_events` が再学習の材料になること。  
> コードとデータ基盤を同時に見るのが、この教材のポイントです。

## [Slide 20] この教材で押さえるべきこと - 35 秒

> MLOps は一つの API ではなく、工程の連結です。  
> BigQuery は保存、特徴量、semantic 検索、ログ分析を横断して支えます。  
> Cloud Run は推論とバッチ学習に加えて lexical 検索基盤も支えます。  
> そして Pub/Sub と Scheduler と Eventarc が再学習ループをつなぎます。  
> この関係が見えれば、この教材の狙いは達成です。

## [Slide 21] まとめ - 30 秒

> このリポジトリは、Cloud Run と BigQuery を中心にした軽量 MLOps の教材です。  
> 重要なのは、推論 API 単体ではなく、Meilisearch の BM25 と BigQuery VECTOR_SEARCH を RRF で束ね、さらに再学習までつなぐパイプライン全体です。  
> そして GCP の各技術は、その各工程を分担しながら一つの運用ループを作っています。  
> 次は実際に `make ops-search` と `make ops-check-retrain` を見ながら、コードと BigQuery を追ってみてください。
