# GCP-ML Vertex 編

## Phase 3 に Vertex AI レイヤを後付けする

**対象**
`/home/ubuntu/repos/study-gcp-mlops/4/study-hybrid-search-vertex` を読む IT エンジニア

**この動画で扱うこと**
- Phase 3 `bq-first` から Phase 4 `vertex` への差分
- Vertex AI プリミティブ (Pipelines / Feature Store / Endpoints / Model Registry / Experiments / Model Monitoring v2 / Vizier) の役割
- Port/Adapter 境界を保ったまま adapter だけ差し替える改修パターン

---

## 前提とゴール

### 前提
- Phase 1 (ML 基礎) / Phase 2 (ハイブリッド検索 Local) / **Phase 3 (GCP Cloud Run + BigQuery)** は受講済
- `Meilisearch BM25 + BigQuery VECTOR_SEARCH + RRF + LightGBM LambdaRank` の設計思想は既知

### この動画のゴール
1. Phase 3 → 4 で **採否が変わる / 変わらない** 項目を言える
2. Vertex AI プリミティブがどの工程を担当するかを整理できる
3. Port/Adapter 境界を保って adapter 差し替えだけで移行する意味を説明できる

> 今回は「Phase 3 を作り直す」ではなく「Phase 3 を **Port/Adapter 経由で継承**して Vertex レイヤを足す」差分工事が主題です。

## この教材の位置付け

**題材**
- 不動産検索 (ドメインは Phase 3 から不変)
- 3 段構成 = Meilisearch BM25 / BigQuery VECTOR_SEARCH / RRF / LightGBM LambdaRank 再ランク **も不変**

**この教材で強調する点**
- Phase 3 のコア (検索 / 特徴量 / 学習ロジック) を **残しつつ** Vertex プリミティブへ移す境界設計
- 採用する / しない Vertex 機能の trade-off
- 実運用で落ちやすい箇所 (traffic_split / VECTOR INDEX / Doppler) の現実解

## 採否が **変わらない** 項目

| 項目 | 理由 |
|---|---|
| Meilisearch on Cloud Run (BM25) | Vertex に BM25 相当なし。Phase 2 / 3 を継続 |
| BigQuery `VECTOR_SEARCH` + VECTOR INDEX | 遅延 p50 < 200ms 実績、IaC 手動 DDL も定着済。Matching Engine 化は別スコープ |
| Dataform `property_features_daily` / assertions | Feature Group のソーステーブルにそのまま使う |
| Pub/Sub → BQ Subscription (`ranking-log` / `search-feedback`) | Vertex 側でも同じ経路でログ蓄積 |
| `monitoring/validate_feature_skew.sql` | Vertex Monitoring 正系後も **バックアップ** として保持 |
| 5 SA 最小権限 + WIF (SA Key 禁止) | Vertex 用 SA を **追加** する形で維持 |
| Port/Adapter + AST/parity テスト | 新 Adapter (`VertexEndpointReranker` / `VertexEndpointEncoder`) を追加するだけで境界が効く |

> Vertex に「相当物が無い / 移行 ROI が低い」ものは意図的に残しています。

## search-api がプロキシになる

### Before (Phase 3)
- `app.entrypoints.api.py::lifespan` で `E5Encoder.load()` と `lgb.Booster(model_file=...)` を **同居ロード**
- Cloud Run メモリ 4Gi、cold start 数秒〜数十秒

### After (Phase 4)
- lifespan は Vertex Endpoint クライアント (`VertexEndpointEncoder` / `VertexEndpointReranker`) を DI するだけ
- Cloud Run メモリ **2Gi**、重い依存 (torch / lightgbm) を API workspace から削除
- ロジック骨格 (`run_search` = RRF → rerank → top-K) は **不変**

```python
# app/src/app/entrypoints/api.py (抜粋)
app.state.encoder_client = VertexEndpointEncoder(
    project_id=settings.project_id,
    location=settings.vertex_location,
    endpoint_id=settings.vertex_encoder_endpoint_id,
)
app.state.reranker_client = VertexEndpointReranker(...)
```

## Port/Adapter 境界の維持

### 新 Port (Protocol)
- `app/src/app/ports/encoder_client.py::EncoderClient` (`embed(text, kind) -> list[float]`)
- `app/src/app/ports/reranker_client.py::RerankerClient` (`predict(instances) -> list[float]`)

### 新 Adapter
- `app/src/app/adapters/vertex_prediction.py`
  - `VertexEndpointEncoder` / `VertexEndpointReranker` — どちらも `aiplatform.Endpoint.predict` を叩く薄い wrapper

### この境界のうれしさ
- `services/ranking.py::run_search` は Port しか見ないので、Local テストでは stub adapter で十分
- `scripts/checks/layers.py::RULES` で Port 側に `google.cloud` が紛れ込まないことを AST で強制

> 「Vertex を入れても pure logic は書き換えない」が守れる理由です。

## CPR カスタムコンテナ 2 種

Vertex AI **Custom Prediction Routine** に従う FastAPI server を 2 本書く。

### encoder (`jobs/src/training/entrypoints/encoder_server.py`)
- `AIP_HEALTH_ROUTE` / `AIP_PREDICT_ROUTE` / `AIP_HTTP_PORT` 準拠
- 起動時 `AIP_STORAGE_URI` (GCS) から ME5 checkpoint を download → `E5Encoder.load`
- 入力 `{"instances": [{"text": "...", "kind": "query"}]}` → 出力 `{"predictions": [[768d]]}`
- **推論ロジックは `common/embeddings/e5_encoder.py` をそのまま import** (Phase 3 と同一)

### reranker (`jobs/src/training/entrypoints/reranker_server.py`)
- 同じく AIP_* 準拠
- 起動時 `model.txt` を GCS から download → `lgb.Booster(model_file=...)`
- 入力 10 次元 float / 出力 scalar score
- 推論時の特徴量作成は API 側でやる前提 (`common/feature_engineering.py::build_ranker_features` と lockstep)

## KFP パイプライン 2 本

### `embed_pipeline` (物件側、日次)
```
load_properties → batch_predict_embeddings → write_embeddings
```
- text_hash 差分で再エンコード対象を絞る (Phase 3 の `embedding-job` のロジックをそのまま移植)
- `ModelBatchPredictOp` で encoder Endpoint を呼ぶ

### `train_pipeline` (週次 + アラート起動)
```
load_features → (vizier 分岐) → train_reranker → evaluate
                                                  └─ dsl.Condition(pass) → register_reranker
```
- `train_reranker` は既存 `jobs/src/training/services/rank_trainer.py::train` を container_component で wrap
- `evaluate` で NDCG@10 閾値チェック、PASS なら Model Registry に登録 + Endpoint カナリアデプロイ

## Model Registry + カナリアデプロイ

### 登録フロー
- `ModelUploadOp` で `property-reranker` に version 追加
- エイリアスは `staging` で初期化
- `scripts/ops/promote.py reranker v<N>` で `production` 付け替え + traffic 100%

### カナリア
- `Model.deploy(traffic_split={"<new>": 10, "<current>": 90})` で 10% から段階投入
- 旧モデルは昇格完了後 `Endpoint.undeploy_model(...)` で剥がしてコスト削減

> Registry は `mlops.training_runs` BQ テーブルと **dual write** で併存。SQL でのアドホック集計を残すため。

## Vertex Experiments と W&B の二重書き込み

### なぜ dual?
- **Vertex Experiments** — Registry / Pipelines / ML Metadata と自動連携、GCP コンソールで lineage 追える
- **W&B** — 可視化、アドホック比較、社外研究で定着 (Phase 1 / 2 から継続)

### 実装
```python
# jobs/src/training/adapters/bigquery_ranker_repository.py::save_run (抜粋)
if experiment_name := os.getenv("VERTEX_EXPERIMENT_NAME"):
    from google.cloud import aiplatform
    aiplatform.init(project=self._project_id, experiment=experiment_name)
    with aiplatform.start_run(run=run_id, resume=True):
        aiplatform.log_params(dict(hyperparams))
        aiplatform.log_metrics({k: float(v) for k, v in metrics.items() ...})
# + BQ mlops.training_runs INSERT は従来通り
```

> Phase 3 の BQ INSERT を **消さず** 足す形です。

## 残す / やめる判断の根拠

### 残したもの
- **Meili + BQ VECTOR_SEARCH + RRF**: Vertex に BM25 相当なし、Matching Engine は遅延に効くが IaC 未対応
- **Dataform**: Feature Group のソーステーブル。SQL 書換は 6 ファイル parity に波及するので凍結
- **`validate_feature_skew.sql`**: Vertex Monitoring 停止時のフォールバック + コスト監査の SQL アドホック可用性

### やめたもの
- **Cloud Run Jobs `training-job` / `embedding-job`**: KFP で代替。`infra/modules/runtime/main.tf` から resource 削除済
- **`app.services.model_store`** (Phase 3 の booster local load): Vertex Endpoint で外部化、3 ファイル削除
- **`/events/retrain` + `CloudRunJobRunner`**: subscriber が Cloud Function に移ったため dead code、削除

## ハマりどころ

### Terraform で **deploy できない** もの
- `google_vertex_ai_endpoint` は空の shell しか宣言できない
- `deployed_models` / `traffic_split` は provider から見て computed、Python SDK (`aiplatform.Model.deploy`) 側で投入
- Terraform には ignore_changes すら書かない (unsupported attribute エラー)

### encoder の初回投入に手順が要る
- reranker は `train_pipeline::register_reranker` で自動化
- **encoder は KFP 外**: `scripts/setup/upload_encoder_assets.py --apply` (ME5 checkpoint を GCS へ) → `scripts/setup/setup_encoder_endpoint.py --apply` (Model.upload + Endpoint.deploy)

### その他
- VECTOR INDEX は **provider 未対応**、`CREATE VECTOR INDEX` の手動 DDL 必須
- Doppler → Cloud Run Secret Manager 経路は **配線していない** (roadmap §9 確定)、`--set-env-vars` 直書き運用
- `infra/variables.tf::oncall_email` は default 無し、`@` validation 付きで placeholder 事故を防ぐ

## この教材で押さえるべきこと

1. Phase 3 の **検索コアはそのまま**、Port/Adapter の adapter だけを Vertex 実装へ差し替え
2. Vertex は **7 コンポーネント** (Pipelines / Feature Store / Endpoints / Registry / Experiments / Monitoring v2 / Vizier) が分担
3. Cloud Run Jobs が KFP に移り、再学習 subscriber が Cloud Function に移った
4. Dual write (Registry ↔ training_runs / Experiments ↔ W&B) で **既存の運用導線を壊さない**
5. CPR コンテナ 2 本と Endpoint 2 本が Phase 4 の新しい学習対象

> Vertex 化は「全部書き直す」ではなく「境界を守って差し替える」工事です。
