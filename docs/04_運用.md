# 運用ガイド

bq-first MLOps パイプラインの **PDCA ループ (deploy-all / destroy-all) / 環境構築の個別 STEP / 定常運用 / インシデント対応 / ロールバック** を扱う。**学習目的の PDCA 高速回転** を最優先のユースケースとし、`make deploy-all` で全 provisioning を一発で済ませ、`make destroy-all` でゼロに戻す運用を §1 で先に押さえる。granular な STEP 1–17 は §2 に残し、PDCA 標準フローで詰まった時だけ参照する位置付け。

詳細設計は `docs/01_仕様と設計.md` と `docs/02_移行ロードマップ.md`、実装カタログは `docs/03_実装カタログ.md` を参照。

> `PROJECT_ID` / `REGION` / `API_SERVICE` / `TRAINING_JOB` / `ARTIFACT_REPO` / `GITHUB_REPO` / `ONCALL_EMAIL` の値は **`env/config/setting.yaml`** が single source of truth。Python (`scripts/_common.py::DEFAULTS`) と Make (awk で読み出し) の両方がここから読む。別プロジェクトを触る場合は yaml を編集するか `PROJECT_ID=xxx REGION=yyy make <target>` で env 上書き。

## 1. PDCA メインフロー (`make deploy-all` ↔ `make destroy-all`)

### 1.0 設計意図

**学習用 dev project (`mlops-dev-a`) で PDCA を最速回転させる**。本リポは試行錯誤の場であり、確認プロンプト / 古いイメージの混入 / 半端な手順は全部排除している:

- **`make deploy-all`** で **provisioning ゼロから image rollout 完了まで一発**。途中で確認は出ない。
- **`make destroy-all`** は **プロンプトなし** で全 Terraform リソースを破壊する (BQ table の `deletion_protection` も自動で剥がす)。
- イメージは **delete-then-push** — Artifact Registry の対象タグを毎回全削除してから push する。stale 由来の "更新したのに動かない" を構造的に排除。
- tfstate バケットと API 有効化は teardown 後も残す → `make deploy-all` を再実行すれば数分で戻る。

```
       ┌────────────────────────┐
   ┌──>│   make deploy-all      │  (約 12-15 分: build × 2 + apply + 90s sleep)
   │   └───────────┬────────────┘
   │               ↓
   │       実 GCP で実験 / 観測 / コード修正
   │               ↓
   │   ┌────────────────────────┐
   │   │   make ops-* で疎通    │  (livez / search / ranking / feedback / label-seed)
   │   └───────────┬────────────┘
   │               ↓
   │     コード変更 → 再度 make deploy-all (image を delete-then-push)
   │               │
   │               ↓
   │   ┌────────────────────────┐
   └───┤   make destroy-all     │  (約 3-5 分: protect 解除 + terraform destroy)
       └────────────────────────┘
```

### 1.1 1 度だけ済ませる前提 (PDCA 周回前の準備)

ここは PDCA ループの**外側**。マシン交換時 / 新規メンバー初回のみ実行する。

```bash
# (a) 前提ツール — uv / terraform / gcloud / make / jq (詳細は §2 STEP 1)
make doctor                                        # 全行 OK で次へ

# (b) Python 依存
make sync

# (c) GCP 認証 + project 固定
gcloud auth application-default login
gcloud config set project mlops-dev-a              # = env/config/setting.yaml::project_id

# (d) env/config/setting.yaml の確認 (要なら編集)
cat env/config/setting.yaml
# project_id / region / github_repo / oncall_email 等が自分の環境で正しいか
```

`make doctor` は `WARN VIRTUAL_ENV=...` を出すことがある (外側で別 venv を `source` 済の場合)。害はないが、混乱回避のため `deactivate` してから make を叩くのを推奨。

### 1.2 PDCA 1 サイクル (典型操作)

```bash
# (1) 立てる
make deploy-all
#   → tf-bootstrap → tf-init → sync-dataform-config → tf-plan → terraform apply
#   → deploy-api-local (Cloud Build + run deploy)
#   → deploy-training-job-local (Cloud Build + run jobs update)
#   約 12-15 分。途中の対話プロンプトなし。

# (2) 疎通
make ops-livez                                     # → {"status":"ok"}
make ops-api-url                                   # → https://search-api-...run.app

# (3) 試したい変更を加える (コード / Terraform / Dataform)
$EDITOR ...

# (4) 再 rollout (image purge → push → revision 更新)
make deploy-all
#   tfstate / バケット / API 有効化はそのまま、差分だけ apply。image は必ず一度 purge して再 push。

# (5) PDCA セッション終了 → 全部消してコスト止める
make destroy-all
#   プロンプトなし。約 3-5 分で terraform destroy 完了。
#   tfstate バケットは残る → 翌日 (1) から再開可能。
```

### 1.3 `make deploy-all` の内訳 (7 step)

`scripts/setup/deploy_all.py` が以下を順に Python module 直 import で実行する。**個別 step を単独 make target で叩いても等価**:

| # | 中身 | 個別 target | 所要時間 (目安) |
|---|---|---|---|
| 1 | API 15 種有効化 + tfstate バケット作成 (冪等) | `make tf-bootstrap` | ~30s (初回)、再実行 ~5s |
| 2 | tfstate バケット preflight + `terraform init` | `make tf-init` | ~10s |
| 3 | `env/config/setting.yaml` → `definitions/workflow_settings.yaml` 生成 | `make sync-dataform-config` | <1s |
| 4 | `terraform plan -out=tfplan` (vars は setting.yaml から) | `make tf-plan` | ~20s |
| 5 | `terraform apply -auto-approve tfplan` (90s の `time_sleep` を含む) | (組込) | ~6-8 分 (初回) / ~2 分 (再 apply) |
| 6 | search-api: AR purge → Cloud Build → `gcloud run deploy` | `make deploy-api-local` | ~3-4 分 |
| 7 | training-job: AR purge → Cloud Build → `gcloud run jobs update` | `make deploy-training-job-local` | ~3-4 分 |

詰まった step は §2 の STEP に対応 (1↔STEP 8、2↔STEP 9 head、4-5↔STEP 9 後半)。

### 1.4 `make destroy-all` の内訳

`scripts/setup/destroy_all.py`:

1. `terraform apply -auto-approve -var=enable_deletion_protection=false -target=<7 BQ tables>` — 7 table (`mlops.training_runs` / `mlops.search_logs` / `mlops.ranking_log` / `mlops.feedback_events` / `mlops.validation_results` / `feature_mart.property_features_daily` / `feature_mart.property_embeddings`) の `deletion_protection` 属性だけを state-flip。`-target` が load-bearing — 半 destroy 状態で SA recreate が fail するのを回避するため、apply 範囲を tables に限定する。
2. `terraform -chdir=infra destroy -auto-approve -var=enable_deletion_protection=false -var=github_repo=... -var=oncall_email=...` (vars は env > setting.yaml)。

**所要時間**: ~3-5 分 (state-flip apply ~30s + destroy ~2-4 分、54 リソース目安)。

**残るもの** (再 deploy のため意図的に保持):
- tfstate バケット `gs://mlops-dev-a-tfstate`
- API enablement (リソース不在時はゼロ課金)

完全クリーンスレートにしたい場合のみ、destroy 後に `gcloud storage rm --recursive gs://mlops-dev-a-tfstate` を手動実行。

### 1.5 PDCA を成立させている技術的保証

PDCA を回す上で「触るたびに違う結果」になると致命的。本リポは以下で再現性を担保:

| 機構 | 場所 | 効果 |
|---|---|---|
| `setting.yaml` を single source of truth に集約 | `env/config/setting.yaml` + `tests/parity/test_dataform_workflow_settings.py` | drift を CI が検知。Make / Python / Dataform yaml が同じ値を見る |
| **delete-then-push** (Artifact Registry purge → Cloud Build push) | `scripts/deploy/{api_local,training_job_local}.py` | stale digest / 古い tag の混入を構造的に排除。"deploy したのに動かない" が起きない |
| **no-prompt** (確認プロンプトなし) | `scripts/setup/destroy_all.py` ほか | PDCA を `make` 1 発で完走、自動化を妨げない |
| placeholder image + `lifecycle.ignore_changes` | `infra/modules/runtime/main.tf` | 初回 apply 時に image が無くても Cloud Run 作成成功、後続 `deploy-*-local` で real image に差し替え可 |
| tfstate / API 有効化を destroy で消さない | `scripts/setup/destroy_all.py` | 翌サイクルの `deploy-all` が即座に開始可能 (Phase 0 を毎回やらない) |
| feature parity invariant (5 ファイル lockstep) | `tests/parity/test_feature_parity_*.py` | 試行錯誤中に schema が崩れて scheduled query が FAIL するのを CI で防ぐ |

### 1.6 PDCA で踏みやすいハマり (修正済)

`make deploy-all` を初回叩いたとき、過去に踏んで **infra/scripts 側で恒久対処済** のハマりを置いておく (再発時にここを起点に追える):

| 症状 | 原因 | 対処箇所 |
|---|---|---|
| `Cannot find metric(s) that match type = "logging.googleapis.com/user/search_api_5xx"` | log-based metric 直後 ~60s は Monitoring API の index が追いつかず alert policy が参照不可 | `infra/modules/monitoring/main.tf` の `time_sleep.wait_for_log_metric_indexing` (90s) + 両 alert policy の `depends_on` |
| `Cannot create a transfer in JURISDICTION_US when destination dataset is located in REGION_ASIA_NORTHEAST_1` | `google_bigquery_data_transfer_config.location` 既定が US で、`asia-northeast1` の dataset と不整合 | `infra/modules/monitoring/main.tf` に `location = var.region`、root から `region` を渡す |
| `Failed to find a valid credential. The field 'version_info' or 'service_account_name' must be specified.` | DTS は `version_info` (user OAuth) か `service_account_name` (SA 代理) のいずれか必須、provider 既定はどちらも付けない | DTS に `service_account_name = var.service_accounts.dataform.email`、`infra/modules/data/main.tf` に `dataform_mlops_editor` を追加 |
| `Image '...search-api:latest' not found` | 初回 apply 時 Artifact Registry が空 | runtime モジュールで `image = "gcr.io/cloudrun/hello"` placeholder + `lifecycle.ignore_changes` |
| `make tf-plan` で `ONCALL_EMAIL is required` (backslash 改行で trailing whitespace 混入) | shell の line continuation 破壊 | setting.yaml が既定値を持つ + Make を 1 行で叩く |
| Cloud Run の `/healthz` が HTML 404 | Knative frontend が literal `/healthz` を予約 | app に `/livez` alias、`make ops-livez` を疎通用に使う |
| `embedding-job` 実行で NOT_FOUND | 旧構成では Job が Terraform 未定義だった | 現在は `infra/modules/runtime/main.tf` で `embedding-job` を Terraform 管理し解消済 |

## 2. 個別 STEP 詳細 (PDCA から外れた手動操作が必要な場合)

`make deploy-all` で完結しない場面 — ローカルだけで API を立てたい (GCP 認証ゼロ) / 1 step だけ手動で叩きたい / CI を有効化したい — の参考用。フローは旧 STEP 1–17 をそのまま残してある。

**STEP 1–6** まではローカル完結 (GCP 認証不要)。お試し / CI 目的はここで終了。
**STEP 7–8** は PDCA 周回開始前の 1 回だけ (※ `make deploy-all` 内部でも実行されるので、deploy-all を回す予定なら個別実行は不要)。
**STEP 9–17** は `make deploy-all` のサブステップを分解したもの + ステージング疎通の中身。

### STEP 1: 前提ツールをインストール

| ツール | 用途 | 最低バージョン |
|---|---|---|
| Python | ランタイム | 3.12 |
| [`uv`](https://docs.astral.sh/uv/) | パッケージ / ワークスペース管理 | 0.4 |
| [`terraform`](https://developer.hashicorp.com/terraform/install) | IaC (STEP 8 以降) | 1.9 |
| [`gcloud`](https://cloud.google.com/sdk/docs/install) CLI | GCP 操作 (STEP 7 以降) | 最新 |
| `make` | ワークフロー一括実行 | 3.8 |
| `jq` | `make ops-check-retrain` の JSON パース | 任意 |
| `docker` | ローカル Dockerfile ビルド確認 | 任意 |

Ubuntu / WSL のインストール例:

```bash
# uv
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env    # PATH を反映 (新ターミナルで不要)

# Terraform (HashiCorp 公式 APT)
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform

# gcloud CLI
sudo apt install google-cloud-cli

# jq (任意)
sudo apt install jq
```

### STEP 2: ツール到達確認

```bash
make doctor
```

全行 `OK` で次へ。`MISS` が出たら STEP 1 に戻る。`python3` のバージョンは `pyproject.toml` の `requires-python = ">=3.12"` を満たせば何でもよい (3.12 / 3.13 / 3.14 いずれも可)。

`WARN VIRTUAL_ENV=... は project 外` が出たら、外側で別の venv を `source` している。uv は repo 直下の `.venv/` を使うため活動中の venv は無視される (害はないが混乱の元)。`deactivate` してから以降の `make` を叩くのを推奨。

### STEP 3: Python 依存を入れる

```bash
make sync
```

uv workspace (`common` / `app` / `jobs`) と `dev` グループ (ruff / mypy / pytest / httpx) を一括導入。`.venv/` が root に作られる (`uv` が自動管理するので手動で activate 不要 — `make` 系は常に `uv run` 経由で解決する)。

**所要時間**: 初回 ~2-3 分 (`sentence-transformers` が引く `torch` ~2GB ほか ML deps の download + 展開)。再実行時はキャッシュヒットで数秒。

### STEP 4: 静的検証 + テスト

```bash
make check
```

ruff / ruff-format / mypy strict / pytest を順に実行。CI と同じ。全緑で次へ。

### STEP 5: 合成データで学習 smoke

```bash
make train-smoke-persist
```

`/tmp/bq-first-smoke-model.txt` に LightGBM LambdaRank モデルが書き出される。BQ/GCS/W&B は触らない。

### STEP 6: API をローカル起動 (GCP 認証ゼロ)

```bash
make api-dev
```

uvicorn が **フォアグラウンドで** `http://127.0.0.1:8000` に bind する (`ENABLE_SEARCH=false` 既定なので `/search` は 503 を返す)。別ターミナルで疎通:

```bash
curl -sSf http://127.0.0.1:8000/healthz
```

/search を local で叩きたい場合は ADC + `ENABLE_SEARCH=true` / `ENCODER_MODEL_DIR=<path>` を入れた別プロセスで起動する (後続 STEP)。
確認後 **Ctrl+C で api-dev を停止** してから STEP 7 へ。GCP にデプロイしない場合はここで終わり。

### STEP 7: GCP 認証 (ADC)

```bash
gcloud auth application-default login      # ブラウザで OAuth
gcloud config set project mlops-dev-a      # 以降の gcloud / bq のデフォルト project
```

### STEP 8: Phase 0 ブートストラップ (プロジェクト新規時のみ、1 回)

Terraform の**外側**で実施する初期設定。API 有効化と tfstate バケット作成を冪等に行う:

```bash
make tf-bootstrap
```

- プロジェクト Owner 相当 (Service Usage Admin + Storage Admin) の権限が必要
- 既存プロジェクトに途中参加なら既に完了しているはずなので再実行しても害なし (冪等)
- 作成されるバケット: `gs://mlops-dev-a-tfstate` (`infra/backend.tf` と一致)
- **所要時間**: 初回 ~30s (15 API 有効化 + バケット作成)、再実行 ~5s

### STEP 9: Terraform apply (インフラ構築)

> **PDCA 標準フローでは `make deploy-all` がここを step 4-5 として内包する**。本 STEP は手動で plan → apply を分けたいとき用。

```bash
make tf-init                                    # バケット存在 + provider (google / time) を preflight でチェック
make tf-plan                                    # GITHUB_REPO / ONCALL_EMAIL は env/config/setting.yaml が既定値
terraform -chdir=infra apply tfplan             # 保存済み plan を apply (対話で yes)
```

別 GitHub repo / 別 oncall を試す場合のみ env で上書き:

```bash
make tf-plan GITHUB_REPO=other/repo ONCALL_EMAIL=oncall@example.com
```

ポイント:
- `GITHUB_REPO` は **GitHub リポジトリのパス**。WIF の `attribute.repository` 制約に焼き込まれる — 間違えると CI が auth できなくなる。setting.yaml の `github_repo` が canonical。
- `ONCALL_EMAIL` は monitoring alert policy の通知先。`@` を含まないと `variables.tf` + `scripts/setup/tf_plan.py` の validation で reject される。setting.yaml の `oncall_email` が canonical。plan 段階では通知送信されないので、練習用途なら自身の address でよい。
- `terraform -chdir=infra apply` を `-var` 無しで直接叩くと対話プロンプトで vars を再入力させられ、`yes` 確認と混同しやすい。**必ず plan を `-out=tfplan` で保存して `apply tfplan` する** (Makefile の `tf-plan` が `-out=tfplan` を付けている)。
- オフラインで syntax チェックだけしたい場合は `make tf-validate` (バックエンド接続なし)。
- `make tf-plan` を **backslash 改行で書かない** (貼り付け時に trailing whitespace が混入すると line continuation が壊れる)。**1 行で叩く** のが安全。
- 初回 apply の所要時間は ~5-8 分 (`time_sleep.wait_for_log_metric_indexing` 90s 含む)。
- 初回は Cloud Run が `gcr.io/cloudrun/hello` placeholder image で起動 (Artifact Registry 空のため)。real image は `make deploy-api-local` / `make deploy-training-job-local` で差し替える (PDCA 標準フローでは `deploy-all` が連続実行)。
- 過去に踏んだ初回 apply のハマり 4 種類 (time_sleep / DTS location / DTS service_account_name / Cloud Run image placeholder) は **§1.6 の表にまとめて記載**。再発時はそちらを参照。

### STEP 10: Secret Manager に値を投入 (実は **skip 可**)

Terraform は secret container (`doppler-service-token` / `wandb-api-key`) だけ作る。本 STEP は値の投入を扱う。

**現状この STEP を skip しても動く** (構造的に Secret Manager は誰も読んでいない):
- Cloud Run deploy は `--set-secrets` を使わず、Doppler 注入は CI workflow が **GitHub Secret** (`secrets.DOPPLER_TOKEN_*`) から行う (Secret Manager 経由ではない)
- `training-job` は `wandb_api_key` が空なら `mode='offline'` に落ちる graceful fallback あり
- `IAM secretAccessor` バインドは存在するが actAs が無いので dead

ただし将来 Doppler / W&B を Secret Manager 経由で配線する移行に備えて手順を残す:

```bash
export DOPPLER_TOKEN="<Doppler の Service Token>"
export WANDB_KEY="<W&B の API Key>"

echo -n "$DOPPLER_TOKEN" | \
  gcloud secrets versions add doppler-service-token --data-file=-
echo -n "$WANDB_KEY" | \
  gcloud secrets versions add wandb-api-key --data-file=-
```

トークン取得元: Doppler は `dashboard.doppler.com` → Projects → Service Tokens、W&B は `wandb.ai/authorize`。

### STEP 11: GitHub Actions Repository variables (CI 経由 deploy をしたい場合のみ)

CI 経由 deploy をしないなら **skip 可** (`make deploy-api-local` で local から代替できる)。CI を有効化する場合は、`terraform -chdir=infra output` で 2 つの値を取り、GitHub の `Settings → Secrets and variables → Actions → Variables` に登録:

```bash
terraform -chdir=infra output workload_identity_provider
terraform -chdir=infra output github_deployer_sa_email
```

| GitHub Variables | 登録する値 |
|---|---|
| `WORKLOAD_IDENTITY_PROVIDER` | `workload_identity_provider` output (`projects/.../locations/global/workloadIdentityPools/github/providers/github-oidc`) |
| `DEPLOYER_SERVICE_ACCOUNT` | `github_deployer_sa_email` output (`sa-github-deployer@mlops-dev-a.iam.gserviceaccount.com`) |
| `ONCALL_EMAIL` | terraform.yml が `-var=oncall_email=...` で参照する |

### STEP 12: Dataform リポジトリの確認

`google_dataform_repository` は Terraform で作成済 (`modules/data/main.tf`)。STEP 9 の apply 後、GCP コンソール (`console.cloud.google.com/bigquery/dataform`) で repository `bq-first` が存在することを確認する。`deploy-dataform.yml` が `definitions/**` 変更時にこのリポジトリへ compilationResults を POST する。

```bash
make ops-runs-recent   # 空でも OK (まだ学習が走っていない初期状態)
```

この時点では `mlops.training_runs` は空。データ投入と初回疎通は STEP 14 以降で行う。

### STEP 14: 不動産データ seed 投入 (ステージング疎通の下準備)

`raw.properties` / `mlops.feedback_events` はインフラ側の骨しか無い (実データはアプリ外注 or seed ツールで投入)。疎通確認だけが目的なら最低 50 件の物件で足りる。

```bash
# 物件 seed (サンプル CSV を raw.properties に load)
bq load --project_id=mlops-dev-a \
  --source_format=CSV --autodetect \
  raw.properties \
  /tmp/properties_sample.csv

# 最低ラインの疎通確認には feedback_events は空でよい (/search の結合が LEFT JOIN)
```

CSV 列は `raw.properties` スキーマ (`property_id / title / description / city / ward / rent / management_fee / layout / area_m2 / age_years / walk_min / pet_ok / created_at / updated_at`) に合わせる。本番投入は別の ETL が受け持つ前提。

### STEP 15: 初回 embedding バッチ実行

`embedding-job` を 1 回回して `feature_mart.property_embeddings` を埋める。初回は全件 encode するので物件数 × ~50ms/件。

```bash
# Cloud Run Jobs が Terraform apply 済の前提
gcloud run jobs execute embedding-job --region asia-northeast1 --wait

# 結果確認 (property_id の件数 = raw.properties 件数、embedding 次元 = 768)
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT COUNT(*) AS n, AVG(ARRAY_LENGTH(embedding)) AS dim
  FROM `mlops-dev-a.feature_mart.property_embeddings`'
```

`dim = 768` / `n > 0` が期待値。`dim` が異なる場合は encoder の checkpoint (multilingual-e5-base) が取り違っている — `ENCODER_MODEL_DIR` 環境変数を確認する。

**所要時間**: 初回 encoder download ~30s + 物件数 × ~50ms/件。50 物件なら ~35s、1,000 物件 ~80s、10,000 物件 ~9 分。`--wait` 付きは完了まで block する。

### STEP 16: VECTOR INDEX 作成 (手動 SQL)

`google_bigquery_vector_index` リソースは 2026-04 時点の Terraform google provider では未対応なので **BQ コンソールまたは bq CLI で手動作成**。小データでは INDEX 無しでも `VECTOR_SEARCH(...)` は brute-force で動くが、物件 10,000 件超で必須:

```bash
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  CREATE OR REPLACE VECTOR INDEX property_embeddings_idx
  ON `mlops-dev-a.feature_mart.property_embeddings`(embedding)
  OPTIONS(
    index_type = "IVF",
    distance_type = "COSINE"
  )
'
```

作成には数分かかる (物件 10k で ~2-5 分、100k で ~10-20 分)。進捗は `INFORMATION_SCHEMA.VECTOR_INDEXES`:

```bash
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT index_name, coverage_percentage, last_refresh_time
  FROM `mlops-dev-a.feature_mart.INFORMATION_SCHEMA.VECTOR_INDEXES`'
```

`coverage_percentage` が 100 に到達したら疎通確認へ。`property_embeddings` を更新するたびに BQ が incremental refresh する (手動再実行不要)。

### STEP 16b: Meilisearch index 同期 (手動初期化)

hybrid 検索では lexical 候補抽出に `meili-search` を使う。`properties_cleaned` を index `properties` へ同期する:

```bash
# meili-search の URL を確認
terraform -chdir=infra output meili_base_url

# 初回同期 (Cloud Run Job 化前の手動実行)
uv run --package jobs meili-sync \
  --project-id mlops-dev-a \
  --table feature_mart.properties_cleaned \
  --meili-base-url "$(terraform -chdir=infra output -raw meili_base_url)" \
  --index properties \
  --require-identity-token
```

同期後は `meili-search` の `/indexes/properties/stats` でドキュメント件数を確認し、`property_embeddings` 件数と大きく乖離していないことを確認する。

### STEP 17: `/search` `/feedback` ステージング疎通 (完了ゲート)

`docs/02_移行ロードマップ.md §10 Phase 4 完了条件` の 5 項目を逐一確認する。**rerank 無し** で `final_rank == lexical_rank` が返ることが合格ライン (booster はまだ load しない):

```bash
# search-api に Phase 4 の env を投入して reload
# 所要時間 ~1 分: Cloud Run revision 再作成 + lifespan で multilingual-e5-base
# の encoder を HuggingFace から pull + SentenceTransformer に load。
# 以降の revision rollout (`make deploy-api-local`) は encoder 層が既存な
# ので ~20s まで短縮。
gcloud run services update search-api --region asia-northeast1 \
  --update-env-vars "ENABLE_SEARCH=true,ENCODER_MODEL_DIR=intfloat/multilingual-e5-base"

# 1) /search 200 応答 + 上位 20 件返却
URL=$(make -s ops-api-url)
TOKEN=$(gcloud auth print-identity-token)
curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"赤羽駅徒歩10分 ペット可","filters":{"max_rent":150000},"top_k":20}' \
  | jq '.results | length'
# → 20 (上限)

# 2-3) final_rank == lexical_rank / score=null (rerank 未配線)
curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"新宿駅","top_k":5}' \
  | jq '.results[] | {property_id, lexical_rank, final_rank, score}'
# → すべての行で lexical_rank == final_rank、score は null

# 4) ranking_log に 1 リクエストあたり候補 100 件分の行が流れる
# (Pub/Sub → BQ Subscription 経由。反映まで数分)
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT request_id, COUNT(*) AS candidates
  FROM `mlops-dev-a.mlops.ranking_log`
  WHERE ts >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 10 MINUTE)
  GROUP BY request_id
  ORDER BY candidates DESC LIMIT 5'
# → candidates = 100 (pool sizeと一致)

# 5) /feedback publish 疎通
REQ_ID=$(curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"品川駅","top_k":1}' | jq -r .request_id)
PROP_ID=$(curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/search" -H 'content-type: application/json' \
  -d '{"query":"品川駅","top_k":1}' | jq -r '.results[0].property_id')
curl -sSf -X POST -H "Authorization: Bearer $TOKEN" \
  "$URL/feedback" -H 'content-type: application/json' \
  -d "{\"request_id\":\"$REQ_ID\",\"property_id\":\"$PROP_ID\",\"action\":\"click\"}"
# → {"accepted": true}

# feedback_events への反映 (数分)
bq query --use_legacy_sql=false --project_id=mlops-dev-a '
  SELECT COUNT(*) AS n
  FROM `mlops-dev-a.mlops.feedback_events`
  WHERE ts >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 10 MINUTE)'
# → n >= 1
```

5 項目すべて PASS でステージング疎通完了。以降は Phase 6 の rerank 組み込み (`/search` に booster を wire して `score` を埋める) に進む。トラブル時は §4.7 を参照。

## 4. インシデント対応

### 3.1 search-api 5xx > 1% アラート発火

1. ログ確認: Cloud Logging で `resource.labels.service_name="search-api" severity>=ERROR`
2. 直近デプロイ確認: `gcloud run revisions list --service search-api --region asia-northeast1 --limit 5`
3. ロールバック: §5 ロールバック早見表の「search-api イメージ」を実行
4. Pub/Sub publish エラーなら API は 200 を返すが `ranking_log` / `feedback_events` 行数が減る — `make ops-search-volume` と `bq query ... mlops.ranking_log` で追跡

### 3.2 p95 レイテンシ > 500ms

- `jsonPayload.latency_ms` が大きい request_id を特定
- モデルサイズ増大が原因なら `--cpu 4 --memory 4Gi` 一時拡張 (コスト影響あり、復旧後戻す)
- `--min-instances 1` を 0 に落とさない (コールドスタートで model load に数秒)

### 3.3 スキュー検出 FAIL (`status = 'FAIL'`)

- どの特徴量か: `make ops-skew-latest` で `feature_name` をチェック
- 推論側ソース (呼び出し元) が変わっていないか → API アクセスログと照合
- 学習分布が古い場合は `make ops-check-retrain` → 条件成立で自動キック / 手動なら `make ops-train-now`
- **しきい値を引き上げる前に必ず原因を特定**

### 3.4 再学習が走らない / 走り過ぎる

- `make ops-check-retrain` を叩き `reasons` を確認
- 閾値は `app/src/app/services/retrain_policy.py` の `NEW_FEEDBACK_ROWS_THRESHOLD=10_000` / `NDCG_DEGRADATION=0.03` / `STALE_DAYS=7`
- Cloud Scheduler: `gcloud scheduler jobs describe check-retrain-daily --location asia-northeast1`
- Eventarc 配送: Cloud Logging で `resource.type="eventarc.googleapis.com/Trigger"` を絞る

### 3.5 Dataform assertion 失敗

- `feature_mart_assertions.property_features_quality` の行を目視
- NULL 率・範囲外はだいたい生データ (`raw.properties`) の upstream 問題
- `property_features_freshness` 失敗 = 過去 2 日分のデータ欠損 → ETL 側を確認

### 3.6 ranker booster ロード失敗 (Phase 6+)

- `readyz` が 503 のまま: `mlops.training_runs` に `finished_at IS NOT NULL` の行が無い → `make ops-train-now`
- rerank を有効化する前は (Phase 4) /readyz は `search_enabled: true` を返しつつ `/search` は `score=null` を返すのが正

### 3.7 `/search` が 503 / 200 空配列

- `readyz` → `search_enabled: false` の場合: `ENABLE_SEARCH=true` と `ENCODER_MODEL_DIR` の両方が env に入っているか確認 (STEP 17)
- `readyz` → `search_enabled: true` なのに `/search` が 503: encoder の lifespan ロードが失敗している。ログで `sentence_transformers` 由来の stack trace を探す。`ENCODER_MODEL_DIR` がコンテナ内に存在するパスか確認
- `/search` が 200 だが `results = []`: (a) `property_embeddings` が空 (STEP 15 を再実行) / (b) VECTOR INDEX が未作成で brute-force でも top_k=0 を返した (サンプルデータが少なすぎる) / (c) フィルタで全部落ちた (`max_rent` 等を外して再実行)
- `ranking_log` に行が流れない: Pub/Sub topic `ranking-log` と BQ Subscription が Terraform 側で作成済みか確認。`gcloud pubsub subscriptions list --filter="topic ~ ranking-log"` で Subscription の `bigquery_config.table` が `mlops.ranking_log` を指しているか

### 3.8 VECTOR INDEX が効かない / coverage が 100% 未満のまま

- `INFORMATION_SCHEMA.VECTOR_INDEXES` で `coverage_percentage` を確認。数時間経っても 100% に満たない場合は `property_embeddings` が絶え間なく更新されている可能性 (embedding-job を一時停止し、INDEX 作成を完了させる)
- `coverage_percentage = 0` のまま: INDEX 作成クエリ自体が失敗している。`INFORMATION_SCHEMA.JOBS` で DDL ジョブのエラーを確認
- 応急: `VECTOR_SEARCH` は INDEX 無しでも動く (brute-force)。物件 10,000 件未満なら体感差は小さい

### 3.9 Phase 10c 移行: 既存プロジェクト上で California リソースを安全に削除する手順

Phase 10c の Terraform 変更を既存プロジェクトに apply する場合、`deletion_protection=true` で保護されていた旧リソースを **事前に手動削除** しないと `terraform apply` が失敗する。新規プロジェクトには無関係。

以下を順に実行 (冪等):

```bash
# 1. California BQ テーブルの deletion_protection を外す
bq update --project_id=mlops-dev-a --no_deletion_protection mlops.predictions_log

# 2. BQ テーブル削除
bq rm -f --project_id=mlops-dev-a -t mlops.predictions_log

# 3. Pub/Sub topic + Subscription 削除
gcloud pubsub subscriptions delete predictions-to-bq || true
gcloud pubsub topics        delete predictions       || true

# 4. Scheduled Query (California skew check) 削除
#    transfer_config 名は `bq ls --transfer_config --location=asia-northeast1` で確認
TC=$(bq ls --transfer_config --location=asia-northeast1 \
    --filter='displayName:california_housing_skew_check' \
    --format='value(name)' | head -1)
[ -n "$TC" ] && bq rm -f --transfer_config "$TC" || true

# 5. この時点で `terraform plan` を流し、差分がゼロに近づいていることを確認
terraform -chdir=infra plan \
  -var="github_repo=<owner>/<name>" \
  -var="oncall_email=<your-email>"

# 6. 問題なければ apply
terraform -chdir=infra apply
```

state rm が必要な場合もある (`terraform state rm module.data.google_bigquery_table.predictions_log` 等) — Terraform が `Cannot destroy resource protected with deletion_protection` を吐いた場合は上記 1–4 を先に完了させる。
