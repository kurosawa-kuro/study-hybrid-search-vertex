# 移行ロードマップ — 不動産ハイブリッド検索を Vertex AI 活用版へ移行

## 0. 前提

- **対象 GCP プロジェクト**: `mlops-dev-a`
- **リージョン**: `asia-northeast1`
- **Python**: 3.12 / **uv ワークスペース**
- **題材**: 不動産ハイブリッド検索 (自由文クエリ + フィルタ → 物件ランキング上位 20 件)。3 段構成 = (1a) Meilisearch BM25、(1b) BigQuery VECTOR_SEARCH、(2) RRF 融合、(3) LightGBM `lambdarank` 再ランク
- **出発点**: 本リポジトリの bq-first 実装 (Phase 1–10d 完了、`make check` 194 tests、`make deploy-all` で実 GCP 完走済)。つまり「ゼロから作る」ではなく「動いている bq-first パイプラインに **Vertex AI レイヤを後付け**する」差分工事
- **移行後の学習パス**: Vertex AI Pipelines (KFP) × **2 本**
  - `embed_pipeline` — 物件テキスト → ME5 encoder (Vertex AI Endpoint) へ Batch Prediction → `feature_mart.property_embeddings` 更新 (日次、差分のみ)
  - `train_pipeline` — ranking_log + feedback_events + features を join → LightGBM LambdaRank 学習 → 評価 → Model Registry 登録 → Endpoint カナリアデプロイ
- **移行後のサービング**: Cloud Run `search-api` (FastAPI) は **ロジック不変のプロキシ**化。Booster / ME5 encoder 実体は持たず、2 つの Vertex AI Endpoint 経由で推論
  - Endpoint A: `property-encoder-endpoint` (ME5 encoder、クエリ側リアルタイム)
  - Endpoint B: `property-reranker-endpoint` (LightGBM Booster、rerank)
- **Feature Store**: Vertex AI Feature Store **オフライン Feature Group のみ** (Online Store は立てない)
- **モデル管理**: Vertex AI Model Registry (`staging` / `production` エイリアス) × 2 モデル。`mlops.training_runs` テーブルは **コスト監査用に継続** (Vertex Experiments と dual 書き込み)
- **実験管理**: Vertex AI Experiments **と** W&B の **二重書き込み**
- **監視**: Vertex AI Model Monitoring v2 を正系。既存 `monitoring/validate_feature_skew.sql` (mean_drift_sigma) は bq-first 互換バックアップとして保持
- **想定総工数**: 約 **11 日** (rerender-free MVP が既に動いているので、OLD 版 vertex-first (10 日) より +1 日 = encoder の Endpoint 化分)

### 0.1 本ロードマップで採否が変わる項目 (bq-first ↔ vertex-first)

| 項目 | bq-first (現状) | vertex-first (移行後) |
|---|---|---|
| 学習オーケストレーション | Cloud Run Jobs `training-job` + `embedding-job` | **Vertex AI Pipelines (KFP)** `embed_pipeline` + `train_pipeline` |
| 特徴量管理 | Dataform `feature_mart.property_features_daily` | Dataform **+ Vertex AI Feature Group** (同 BQ テーブルを wrap) |
| 再ランク推論 | `search-api` lifespan で `lgb.Booster` を GCS から load | **Vertex AI Endpoint** `property-reranker-endpoint` (CPR 準拠 custom container) |
| 埋め込み (クエリ側) | `search-api` lifespan で sentence-transformers をロード | **Vertex AI Endpoint** `property-encoder-endpoint` (CPR 準拠 custom container) |
| 埋め込み (物件側バッチ) | `embedding-job` Cloud Run Job | `embed_pipeline` KFP の Vertex AI Batch Prediction ステップ |
| モデルレジストリ | GCS パス命名規則 + `mlops.training_runs` テーブル | **Vertex AI Model Registry** (staging/production エイリアス) + `training_runs` は dual 書き込みで残す |
| 監視 | Scheduled Query `property_feature_skew_check` (mean_drift_sigma) | **Vertex AI Model Monitoring v2** (FeatureDriftSpec + PredictionOutputDriftSpec)。既存 SQL はバックアップ |
| 実験管理 | W&B | **Vertex AI Experiments** + W&B (二重書き込み) |
| ハイパラ探索 | 手動 (`rank_trainer.DEFAULT_PARAMS`) | **Vertex AI Vizier** (`enable_tuning=True` で pipeline ON) |
| 再学習キック | Cloud Scheduler → `/jobs/check-retrain` → Pub/Sub `retrain-trigger` → Eventarc → Cloud Run Job | 同経路を **Cloud Function → `PipelineJob.submit()`** に差し替え。Monitoring アラート経由の Eventarc トリガーを追加 |
| サービング SA 要件 | `sa-api` は Pub/Sub publisher + BQ reader | `sa-api` に **`aiplatform.user`** を追加 (Endpoint 呼び出し用) |
| Cloud Run メモリ | `search-api` 4Gi (ME5 1.1GB + Booster + overhead) | **2Gi に削減可能** (encoder / Booster が外部化、`--min-instances 1 → 0` も選択肢) |

### 0.2 本ロードマップで採否が**変わらない**項目 (継続採用)

- ❌ BQML (LambdaRank 非対応、そもそも Vertex AI 経由になる)
- ⚪ **BigQuery VECTOR_SEARCH + VECTOR INDEX** — Vertex AI Vector Search (Matching Engine) への移行は**別スコープ**。BQ VECTOR_SEARCH で十分な遅延 (p50 < 200ms 目安) が出ており、IaC 未対応の手動 DDL 運用も定着済。コストと複雑度の trade-off で据え置き
- ⚪ **Meilisearch Cloud Run Service `meili-search`** — Vertex に BM25 相当なし。Elasticsearch / Meilisearch / Algolia のいずれかが必要。現行 Meilisearch Cloud Run 構成を継続
- ⚪ **Dataform `property_features_daily` / `properties_cleaned`** — Feature Group のソーステーブルとしてそのまま。SQL 式を書き換えると feature parity invariant (下記 §1.4) に波及するので凍結
- ⚪ **Pub/Sub → BQ Subscription** (`ranking-log` / `search-feedback`) — Vertex 側でも同じ経路でログ蓄積
- ⚪ **`monitoring/validate_feature_skew.sql`** — Model Monitoring の正系化後も、コスト監査と「Vertex 停止時のフォールバック」用に保持
- ⚪ **5 SA 最小権限分離 + WIF (SA Key 禁止)** — Vertex 用 SA (`sa-pipeline` / `sa-endpoint-*` / `sa-pipeline-trigger`) を**追加**する形で最小権限を維持
- ⚪ **Port / Adapter + composition root + AST/parity テスト** — 既存アーキテクチャは不変。新しい Adapter (`VertexEndpointReranker` / `VertexEndpointEncoder`) を追加するだけで既存 Port (`CandidateRetriever` / `ModelStore`) に収まる

### Phase 3: ME5 Encoder カスタム CPR コンテナ + Vertex AI Endpoint (1.5 日)

**目的**: クエリ時 / バッチ時の両方で呼ばれる ME5 encoder を Vertex AI Endpoint に移し、`search-api` から 1.1GB の sentence-transformers 依存を剥がす。

**作業項目**:

1. **encoder CPR コンテナ** (`jobs/containers/encoder/`):
   - `Dockerfile` — python:3.12-slim + `torch==2.*+cpu` + `sentence-transformers>=3.0` + `fastapi` + `uvicorn`
   - `jobs/src/training/entrypoints/encoder_server.py`:
     - FastAPI で `GET /health` (AIP_HEALTH_ROUTE) / `POST /predict` (AIP_PREDICT_ROUTE) / `AIP_HTTP_PORT` (既定 8080) 準拠
     - 起動時に `AIP_STORAGE_URI` (GCS) から `multilingual-e5-base/` を `sentence_transformers.SentenceTransformer.from_pretrained` でロード
     - `POST /predict` 入力: `{"instances": [{"text": "...", "kind": "query"}, ...]}`
     - 出力: `{"predictions": [[768次元], ...]}`
     - prefix は server 内部で付与 (`kind="query"` → `"query: "`, `kind="passage"` → `"passage: "`) — bq-first `E5Encoder` の prefix 規約をそのまま継承
   - 推論ロジックは `common/src/common/embeddings/e5_encoder.py` を **そのまま import**

2. **Model Registry 登録 + Endpoint 作成**:
   - Artifact Registry に image push (`deploy-encoder-image.yml` を Phase 10 で作る、当面は `make` 経由手動)
   - GCS にモデル assets を配置 (`gs://mlops-dev-a-models/encoders/multilingual-e5-base/v1/`)
   - `aiplatform.Model.upload` で Registry 登録 (`serving_container_image_uri` = 上記 AR image、`artifact_uri` = 上記 GCS パス、`serving_container_ports=[8080]`、`serving_container_predict_route="/predict"`、`serving_container_health_route="/health"`)
   - `aiplatform.Endpoint.create(display_name="property-encoder-endpoint")`
   - `model.deploy(endpoint=..., machine_type="n1-standard-2", min_replica_count=1, max_replica_count=3, traffic_percentage=100, service_account=<sa-endpoint-encoder>)`
   - Endpoint は `infra/modules/vertex/main.tf` の `google_vertex_ai_endpoint.encoder` で空の Endpoint を宣言、**deploy は Python SDK で実施** (Terraform での deploy ネストは未成熟)

   - `entrypoints/api.py` lifespan: `E5Encoder` ロード経路を削除、`VertexEndpointEncoder` を DI
   - `services/ranking.py` で `encoder_port.embed(query, kind="query")` 呼出に差替 (既存シグネチャ維持)
   - `app/Dockerfile` + `app/pyproject.toml` から `torch` / `sentence-transformers` を削除、`google-cloud-aiplatform>=1.50` 追加
   - Cloud Run 設定を `--memory 2Gi` に変更 (Booster はまだ lifespan に残っている状態なので 4Gi → 2Gi は Phase 6 完了後、本 Phase では 3Gi 暫定)

4. **`embed_pipeline`** 手動 submit → 物件テキストを encoder Endpoint に Batch Prediction → `feature_mart.property_embeddings` に書込
5. **VECTOR INDEX** を手動 DDL で再生成 (`docs/04_運用.md §1 STEP 16`、bq-first と同一運用)
6. Cloud Run プロキシ `search-api` に `/search` を IAM トークン付きで投げる → encoder Endpoint (query 側) → Meilisearch + BQ VECTOR_SEARCH → RRF → reranker Endpoint → top 20 返却
7. `mlops.ranking_log` に行、`mlops.feedback_events` に `/feedback` 送信結果が積まれる
8. **`train_pipeline`** 手動 submit (`--enable-tuning`) → **Vizier** 8 trial → best params で LightGBM LambdaRank 学習 → **Vertex Experiments** に全 trial + final run、**W&B** にも同 run (dual) → Evaluation (NDCG@10 閾値クリア) → **Model Registry** v1 登録 (`staging` alias)
9. `register_reranker` が reranker Endpoint にカナリアデプロイ (新 10% / 旧 90%)
10. `scripts/ops/promote.py reranker v1` で `production` エイリアス切替 + 100% トラフィック
11. **Vertex AI Model Monitoring v2** が日次実行され、7 特徴量の drift と予測ドリフトが評価される → `mlops.model_monitoring_alerts` に結果行
12. 模擬ドリフトで Monitoring アラート発火 → Pub/Sub → Eventarc → Cloud Function → `train_pipeline` 再起動 (自己完結 CT ループ)
13. **Vertex ML Metadata** に全工程のアーティファクト系譜が自動記録される (KFP 実行時に自動)
14. **PipelineJobSchedule** が `embed_pipeline` 日次 + `train_pipeline` 週次で起動し、放置できる
15. `make destroy-all` で全リソース破壊 (Feature Group / Endpoint / Pipeline templates / Cloud Function / Eventarc 含む 80+ リソース)、tfstate バケットは保持
16. `make deploy-all` で 2 周目完走 (WIF / Endpoint / Registry の orphan handling が効く)

## 9. 注記

- **BigQuery VECTOR_SEARCH の Vertex AI Vector Search への置換は本ロードマップの範囲外**。別途検討が必要ならば `docs/05_問題点.md` に案件を書き起こすこと。Matching Engine 化は遅延短縮に効くが、IaC 対応 / Index 更新頻度 / 費用 の 3 点で追加工数 3〜5 日が見込まれる
- **`training_runs` テーブルは削除しない**。Vertex Experiments が正系になっても、BQ 側は SQL でアドホック集計できるコスト監査面で有利。`save_run` が両方に書き込むことで parity を取る
- **Cloud Run `--min-instances 1 → 0` はオプショナル**。Phase 6 完了後、search-api コンテナサイズが激減するため cold start が数秒内に収まり `0` 化も選択肢になる。`CLAUDE.md` の非負制約表では引き続き `1` を既定とし、運用で変更する場合は必ず User に flag する
- **Doppler 連携は本ロードマップでも配線しない** (bq-first と同じ扱い)。Secret Manager 容器は作るが `--set-secrets` 未使用。Vertex 化で必要性は変わらない
- **uv workspace 拡張**: `pipelines/` 追加でワークスペース membership は `app` / `common` / `jobs` / `pipelines` の 4 つ。`make sync` / `make check` はそのまま動く (自動発見)
- **移行期間中の dual running は不可**。Phase 6 で search-api の `/search` が Booster 直呼び出しから Endpoint 呼出に差し替わった瞬間、bq-first 経路は使えなくなる。ロールバックは Cloud Run revision で前バージョンに戻す形で行う (GCS + `training_runs` は保持されているので再学習せずに即復旧可)
